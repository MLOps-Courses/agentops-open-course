# 6.0. Platform

You have built, evaluated, and secured the Ops Copilot. It still runs on your laptop. This chapter turns it into a real **Kubernetes workload** with [kagent](https://kagent.dev/), a CNCF project that makes agents first-class citizens of a cluster. Everything stays local on the k3d cluster from [1.4. Kubernetes](../1. Setup/1.4. Kubernetes.md) — no cloud account required.

## Why run an agent on Kubernetes?

Operating an agent means far more than running a process. It needs to restart when it crashes, scale under load, roll out new versions safely, keep secrets out of source control, and expose a stable network address. Kubernetes already solves all of that for ordinary services, and an agent is — at the deployment layer — just a service that speaks A2A over HTTP.

Kubernetes is also where the Ops Copilot _belongs_: it is an on-call agent for platform engineers, so running it on the same substrate it reasons about closes the loop. [Chapter 7](../7. Observability/) then traces and monitors it there.

## What is kagent?

[kagent](https://kagent.dev/) brings agents to Kubernetes as **custom resources**. Instead of hand-writing Deployments, Services, and ConfigMaps, you declare a few high-level objects — an agent, its model, its tools — and a controller reconciles them into the underlying workloads.

kagent has four moving parts:

1. A **controller** that watches the custom resources and provisions the Kubernetes objects (Deployment, Service) they imply.
1. A set of **CRDs** under the `kagent.dev` API group — `Agent`, `ModelConfig`, `RemoteMCPServer`, and more — that form the declarative API you write.
1. A **web UI** for browsing agents, invoking them, and inspecting runs.
1. An **engine built on Google ADK** — the same framework you used in [Chapter 2](../2. Agents/) — so the runtime semantics you already know carry over. The engine depends on `google.golang.org/adk` (Go) and the `kagent-adk` package (Python).

## Which custom resources does this course use?

You will meet a handful across the next pages. They live under one API group but carry **mixed API versions**, so the exact `apiVersion` matters per resource:

| Resource              | apiVersion            | Covered in                       |
| --------------------- | --------------------- | -------------------------------- |
| `Agent` (`type: BYO`) | `kagent.dev/v1alpha2` | [6.3](./6.3. Platform Agents.md) |
| `ModelConfig`         | `kagent.dev/v1alpha2` | [6.3](./6.3. Platform Agents.md) |
| `RemoteMCPServer`     | `kagent.dev/v1alpha2` | [6.4](./6.4. Platform Tools.md)  |
| `ToolServer`          | `kagent.dev/v1alpha1` | [6.4](./6.4. Platform Tools.md)  |

The version split is a direct consequence of kagent's Sandbox maturity: the API is still stabilizing, and different resources graduated at different times.

!!! info "CNCF project"

    kagent is a [CNCF Sandbox](https://www.cncf.io/projects/kagent/) project (accepted 2025-05-22), Apache-2.0 licensed. Sandbox is the earliest CNCF maturity tier: expect fast-moving, `v1alpha*` APIs. This course pins `v0.10.0-beta3` and treats that churn as a versioning lesson in itself — always re-verify versions and CRD fields at build time.
