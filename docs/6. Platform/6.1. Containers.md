# 6.1. Containers

Before kagent can run the Ops Copilot, it needs an **image**: an immutable, self-contained artifact that carries the agent's code, its pinned runtime, its dependencies, and (in this course) the bundled dataset. The image is built with a multi-stage Dockerfile: a minimal, non-root image that serves the agent over A2A on port 8080.

## What contract must the image satisfy?

kagent's `Agent` with `type: BYO` (Chapter [6.3](./6.3. Platform Agents.md)) deploys _your_ image and expects it to behave like any A2A service:

- Serve over **A2A on port 8080**, listening on `0.0.0.0` so the cluster can reach it.
- Publish an **agent card** at the well-known path `/.well-known/agent-card.json`, which kagent's readiness probe fetches.
- Run as a **non-root** user, the baseline for a workload you would put on a cluster.

The image meets this contract.

## How do you build the agent image?

The image build uses a multi-stage [`Dockerfile`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/agents/python/Dockerfile). A first `uv` stage resolves the locked dependencies into a self-contained virtualenv; a slim runtime stage copies only that venv, the source, and the dataset.

```dockerfile
# --- Stage 1: resolve dependencies into /app/.venv with uv ---
FROM ghcr.io/astral-sh/uv:python3.13-bookworm-slim AS build

ENV UV_COMPILE_BYTECODE=1 \
    UV_LINK_MODE=copy \
    UV_PYTHON_DOWNLOADS=never \
    UV_PROJECT_ENVIRONMENT=/app/.venv \
    VIRTUAL_ENV=/app/.venv

WORKDIR /app

# Locked runtime deps only; --no-install-project runs the agent from source.
COPY python/pyproject.toml python/uv.lock ./
RUN uv sync --frozen --no-dev --no-install-project

# A2A serving needs google-adk's `a2a` extra (deployment-only, kept out of pyproject).
RUN uv pip install "a2a-sdk>=0.3.4,<0.4"
```

Build it from the repo root, where the context is `agents/` so both `python/` and the shared `data/` are visible:

```bash
docker build -f agents/python/Dockerfile \
  -t k3d-registry.localhost:5050/agentops-agent:latest agents
docker push k3d-registry.localhost:5050/agentops-agent:latest
```

!!! note "The PII guardrail adds weight to the image"

    The always-on PII redaction from [4.5. Guardrails](../4. Quality/4.5. Guardrails.md) is a **runtime**
    dependency (Presidio + the small `en_core_web_sm` spaCy model), so the image carries it — a deliberate
    trade-off of size for safety. The dev-only eval tools (MLflow, garak) are *not* in the image: the
    Dockerfile builds with `--no-dev`. If image size is critical, move the PII stack to an optional
    dependency group and enable the callback only where you need it.

## Why not a distroless base?

The image aims for the "distroless spirit" but cannot use `gcr.io/distroless/python3` directly: that base tracks Debian's **Python 3.11**, while the agent requires **3.13** (see the project's `requires-python`). The Dockerfile explains the choice inline:

```dockerfile
# --- Stage 2: slim, non-root runtime ---
FROM python:3.13-slim-bookworm AS runtime

# Run as an unprivileged user (uid/gid 10001); the mock actions write to the dataset.
RUN groupadd --system --gid 10001 app \
 && useradd --system --uid 10001 --gid app --no-create-home app

# ...

USER app
EXPOSE 8080

# Serve A2A on :8080 (0.0.0.0 so kagent and the ingress can reach it).
ENTRYPOINT ["uvicorn", "server:app", "--host", "0.0.0.0", "--port", "8080"]
```

`python:3.13-slim-bookworm` is the closest match: no build toolchain, non-root, and small. In production, pin both bases by `@sha256` digest for reproducible, auditable builds.

## How does the image serve A2A?

The Dockerfile bakes in a tiny launcher that wraps the agent with ADK's `to_a2a`, which auto-builds the agent card from the agent object — so there is no `agent.json` to maintain:

```python
"""Serve the Ops Copilot over A2A (kagent BYO contract, Chapter 6.3)."""

import os

from google.adk.a2a.utils.agent_to_a2a import to_a2a

from agent.agent import root_agent

# A2A Starlette app; host/port only shape the advertised agent-card URL.
app = to_a2a(root_agent, host=os.environ.get("A2A_HOST", "0.0.0.0"), port=int(os.environ.get("PORT", "8080")))
```

## Where does the image go?

The image pushes to the **k3d local registry** at `k3d-registry.localhost:5050`, using the image name `agentops-agent`. That is the address the `Agent` custom resource references in [6.3](./6.3. Platform Agents.md).

## How do you scan the image before shipping?

Do not push an unscanned image. As covered in [4.6. Security](../4. Quality/4.6. Security.md), run [Trivy](https://trivy.dev/) against the built image to surface OS and dependency vulnerabilities:

```bash
trivy image k3d-registry.localhost:5050/agentops-agent:latest
```

Minimal, non-root images already shrink the attack surface; scanning catches what is left. Wire this into CI so a vulnerable image never reaches the cluster.
