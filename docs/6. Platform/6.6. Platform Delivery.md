# 6.6. Platform Delivery

Every piece of the platform now exists: an image, the kagent control plane, an `Agent`, a `ModelConfig`, tool CRDs, and the gateway. This page ties them into a repeatable **delivery loop** — the inner loop you run while developing, and the outer loop CI and GitOps run to ship. Manifests and [`infra/helmfile.yaml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/infra/helmfile.yaml) drive it.

## What is the inner loop?

The inner loop is build → push → apply, on repeat, as you change code. The agent's image is built with the Dockerfile from [6.1](./6.1. Containers.md), pushed to the k3d registry, and the manifests are applied. Run it from `infra/`:

```bash
docker build -f ../agents/python/Dockerfile \
  -t k3d-registry.localhost:5050/agentops-agent:latest ../agents
docker push k3d-registry.localhost:5050/agentops-agent:latest
kubectl apply -f kagent/modelconfig.yaml -f kagent/agent.yaml -f k8s/ingress.yaml
```

Note which manifests this applies: the `ModelConfig`, `Agent`, and ingress (plus the namespace on first run). It deliberately leaves out `kagent/toolserver.yaml` — the `RemoteMCPServer` is applied alongside the agentgateway data plane it points at ([6.4](./6.4. Platform Tools.md)/[6.5](./6.5. Platform Gateway.md)), not in the build-to-deploy inner loop.

## How is the platform itself installed and versioned?

Application manifests (`Agent`, `ModelConfig`, ingress) are one layer; the **control plane** is another. kagent is installed declaratively with helmfile ([6.2](./6.2. Platform Install.md)), which pins both charts:

```yaml
releases:
  - name: kagent-crds
    namespace: kagent
    chart: oci://ghcr.io/kagent-dev/kagent/helm/kagent-crds
    version: 0.10.0-beta3
    createNamespace: true

  - name: kagent
    namespace: kagent
    chart: oci://ghcr.io/kagent-dev/kagent/helm/kagent
    version: 0.10.0-beta3
    createNamespace: true
    needs:
      - kagent/kagent-crds
```

Separating the two layers is deliberate GitOps hygiene: the control plane changes on its own (pinned, infrequent) cadence via `helmfile apply`, while the agent iterates fast via the build-push-apply loop above. Both are declarative files in Git, so the cluster state is reproducible from the repository.

## What is the full quickstart?

The [`infra/README.md`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/infra/README.md) captures the end-to-end path — cluster, namespace, secret, control plane, agent:

```bash
dot cluster start                                   # shared local k3d cluster
kubectl apply -f k8s/namespace.yaml
kubectl -n agentops create secret generic kagent-gemini \
  --from-literal=GOOGLE_API_KEY="$GOOGLE_API_KEY"
helmfile apply                                      # or: kagent install --profile demo
docker build -f ../agents/python/Dockerfile \
  -t k3d-registry.localhost:5050/agentops-agent:latest ../agents && \
  docker push k3d-registry.localhost:5050/agentops-agent:latest && \
  kubectl apply -f kagent/modelconfig.yaml -f kagent/agent.yaml -f k8s/ingress.yaml  # build + deploy
```

Each command maps to a page in this chapter: the cluster ([1.3](../1. Setup/1.3. Kubernetes.md)), the namespace and Secret ([6.3](./6.3. Platform Agents.md)), the control plane ([6.2](./6.2. Platform Install.md)), and the build-and-deploy loop ([6.1](./6.1. Containers.md) + this page).

## How does this map to CI/CD?

The local loop _is_ the pipeline, run by a machine instead of you. In CI:

1. **Build and scan** the image (Dockerfile), failing on Trivy findings ([4.6](../4. Quality/4.6. Security.md)).
1. **Push** the versioned image to the registry.
1. **Apply** the manifests — `helmfile apply` for the control plane, `kubectl apply` for the agent — into the cluster.

Because both layers are declarative files under version control, a GitOps controller can watch the repository and reconcile the cluster to match, giving you auditable, roll-back-able deployments without manual `kubectl`.

!!! warning "Pin everything, re-verify at build time"

    The delivery files pin exact versions — kagent `v0.10.0-beta3`, agentgateway `v1.3.1`. These projects move fast and their APIs are still `v1alpha*`, so re-verify the pinned versions and CRD fields when you build. Reproducible delivery depends on those pins, not on `latest`.
