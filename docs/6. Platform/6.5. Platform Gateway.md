# 6.5. Platform Gateway

In [Chapter 5](../5. Gateway/) you ran [agentgateway](https://aaif.io/projects/agentgateway/) locally as a single binary. On the platform, the same gateway would become an **in-cluster data plane**: the Service that MCP, A2A, and model traffic flow through, and the target the tool CRD from [6.4](./6.4. Platform Tools.md) points at. The concrete thing this page ships is the **ingress** that exposes your deployed agent; standing the gateway up in-cluster is an advanced step this course explains but leaves for you to complete.

!!! warning "This course does not ship an in-cluster gateway"

    The local `agentgateway/config.yaml` launches the Ops Copilot's MCP server over stdio with a host-local `uv run --directory ../agents/python`, which cannot run inside the cluster. A working in-cluster gateway therefore has to co-locate the MCP source (for example an agentgateway sidecar in the agent pod, or an image bundling both). The reference `RemoteMCPServer` in [6.4](./6.4. Platform Tools.md) is a template for _once that Service exists_; the deployed BYO agent already carries its tools in-process and does not need it to work.

## Why would you run the gateway inside the cluster?

The gateway is where connectivity and security policy live — one place to secure, observe, and rate-limit every MCP, A2A, and LLM call. On your laptop it fronted a stdio MCP server ([5.2. MCP Gateway](../5. Gateway/5.2. MCP Gateway.md)). In the cluster it would do the same job for other agents and workloads, so the security posture you built in Chapter 5 carries over unchanged rather than being re-implemented per workload. kagent itself runs agentgateway as its own data plane, so this is the intended integration, not a workaround — it is simply beyond the scope of the reference deploy.

## How would the deployed agent reach the gateway?

Through a Kubernetes Service. The `RemoteMCPServer` from [6.4](./6.4. Platform Tools.md) hard-codes the gateway's in-cluster DNS name:

```yaml
url: http://agentgateway.agentops.svc.cluster.local:3000/mcp
protocol: STREAMABLE_HTTP
```

That would resolve to a Service named `agentgateway` in the `agentops` namespace on port 3000 — the same port the local `agentgateway/config.yaml` binds. `/mcp` is the gateway's streamable-HTTP MCP endpoint (`/sse` with `protocol: SSE` is the alternative). So an agent calls the gateway, and the gateway calls the actual tools — the CRD only records the address, which is why it stays inert until you deploy that Service.

## What traffic does the gateway front?

The gateway is AI-native: it proxies the three protocols an agent platform relies on, each added by chapter in `infra/agentgateway/config.yaml`:

- **MCP** — the Ops Copilot's tools, multiplexed and secured ([5.2](../5. Gateway/5.2. MCP Gateway.md)). This is the active path, the one the `RemoteMCPServer` targets.
- **A2A** — agent-to-agent routing via the `a2a` route policy ([5.3](../5. Gateway/5.3. A2A Gateway.md)).
- **Model / LLM** — pointing the agent's model endpoint at the gateway's LLM backend to reach any provider, including local Ollama ([5.4](../5. Gateway/5.4. Model Gateway.md)).

Routing all three through one data plane means one policy point for auth, rate limiting, and prompt guard ([5.5](../5. Gateway/5.5. Gateway Security.md)) instead of per-workload configuration.

## How is the agent exposed to you?

The agent's own A2A endpoint is reached through a **Traefik ingress** (k3d's default), which routes a `*.localhost` host to the Service kagent generated. From [`infra/k8s/ingress.yaml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/infra/k8s/ingress.yaml):

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: agentops-agent
  namespace: agentops
spec:
  rules:
    - host: agentops.localhost
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: agentops-agent
                port:
                  number: 8080
```

The backend `agentops-agent` on port 8080 is the Deployment and Service kagent provisioned from the `Agent` CR ([6.3](./6.3. Platform Agents.md)) — not duplicated here. After applying it, the agent answers at:

```bash
curl http://agentops.localhost:8080/.well-known/agent-card.json
```

## Local gateway or in-cluster gateway — which is which?

Both use the same binary and the same config shape; only the deployment differs.

1. **Local (Chapter 5)** — `agentgateway -f config.yaml` on your machine, admin UI on `:15000`, for developing and testing policy fast. This is the path the course runs end-to-end.
1. **In-cluster (advanced)** — the gateway as a Service in the `agentops` namespace, fronting tools and reached by DNS. The pattern is above; wiring the MCP source into the cluster is the exercise left for you.

Develop and validate policy locally, then — when you take the platform further — run the identical configuration in-cluster as a shared data plane.
