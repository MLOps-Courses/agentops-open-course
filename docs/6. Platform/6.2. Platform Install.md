# 6.2. Platform Install

With a cluster running ([1.3. Kubernetes](../1. Setup/1.3. Kubernetes.md)) and an image built ([6.1](./6.1. Containers.md)), the next step is to install the kagent control plane: the controller and CRDs that let you declare agents as custom resources. There are two supported paths — a one-command CLI and pinned Helm charts.

## Does the course's k3d cluster work, or do I need kind?

k3d works. kagent's own repository documents a **kind** cluster, but k3d runs the same certified Kubernetes and is API-compatible, so every command and manifest here works on either. This course uses the shared local k3d cluster as its standard:

```bash
dot cluster start   # shared local k3d cluster (or: k3d cluster create local)
```

## How do I install kagent the quick way?

The `kagent` CLI wraps Helm behind a single command. The `demo` profile installs the control plane and preloads a set of sample agents and tools so you can explore the web UI immediately:

```bash
kagent install --profile demo
```

Use `--profile minimal` for a bare control plane with no samples. The CLI auto-detects provider API keys from your environment, which is convenient for a quick trial.

## How do I install kagent reproducibly with Helm?

For anything you want version-controlled, install the two Helm charts directly. The course keeps them in [`infra/helmfile.yaml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/infra/helmfile.yaml) so the versions are pinned and re-runnable:

```yaml
releases:
  - name: kagent-crds
    namespace: kagent
    chart: oci://ghcr.io/kagent-dev/kagent/helm/kagent-crds
    version: 0.10.0-beta3
    createNamespace: true

  - name: kagent
    namespace: kagent
    chart: oci://ghcr.io/kagent-dev/kagent/helm/kagent
    version: 0.10.0-beta3
    createNamespace: true
    needs:
      - kagent/kagent-crds
```

Apply it with one command:

```bash
helmfile apply
```

## Why are the CRDs a separate chart?

kagent ships **two** charts on purpose. `kagent-crds` installs only the custom resource definitions; `kagent` installs the controller and depends on the CRDs being present first (the `needs` clause enforces that order). Splitting them lets you upgrade the controller without racing CRD changes, and lets cluster operators who manage CRDs separately install them on their own schedule. The charts also pull in the bundled **kmcp** subproject, which manages `MCPServer` resources ([6.4](./6.4. Platform Tools.md)).

## CLI or Helm — which should I use?

Both end at the same place; the CLI simply wraps Helm.

- **`kagent install --profile demo`** — fastest for a first look and for exploring the UI with preloaded samples.
- **helmfile / Helm** — declarative, pinned, and reproducible; the right choice for CI and GitOps ([6.6](./6.6. Platform Delivery.md)).

This course leads with helmfile because pinned versions are the whole point of the exercise, but either gets you a working control plane.

## How do I check the install worked?

The controller runs in the `kagent` namespace. Confirm its pods are ready and the CRDs are registered:

```bash
kubectl -n kagent get pods
kubectl get crds | grep kagent.dev
```

You should see the controller pod `Running` and the `kagent.dev` CRDs (`agents`, `modelconfigs`, `remotemcpservers`, `toolservers`, …) listed.

!!! warning "Beta software — pin and re-verify"

    kagent is `v0.10.0-beta3`, a Sandbox-stage project with `v1alpha*` APIs that change between releases. Always **pin** the chart version (as `infra/helmfile.yaml` does) and **re-verify** the CRD fields against the release you install before relying on any manifest in this chapter. Treat the version churn as a first-class part of operating early-stage software, not an afterthought.
