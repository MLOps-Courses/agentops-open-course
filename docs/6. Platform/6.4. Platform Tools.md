# 6.4. Platform Tools

Your BYO Ops Copilot already carries its tools **in-process** — the function tools from [Chapter 3](../3. Capabilities/) are compiled into the image. This page is about the platform's _other_ way of providing tools: registering standalone **MCP servers** as kagent custom resources so that agents — yours or others kagent runs — can discover and call them, exactly as in [3.3. MCP](../3. Capabilities/3.3. MCP.md). Two CRDs do this — pick by transport.

!!! note "Illustrative, not part of the running deploy"

    The deployed agent does not depend on anything below to function — it has its tools already. The `RemoteMCPServer` here shows _how_ you would expose the Ops Copilot's tools to the platform through agentgateway; standing that gateway up in-cluster is an advanced exercise this course leaves open (see [6.5](./6.5. Platform Gateway.md)). Treat these manifests as templates.

## How does a deployed agent get its tools?

You register an MCP server as a kagent custom resource, and the agent picks up its tools. There are two CRDs, differing only in _how_ the MCP server is reached:

- **`RemoteMCPServer`** (`kagent.dev/v1alpha2`) — a remote MCP endpoint over HTTP or SSE. kagent connects to a URL you provide.
- **`ToolServer`** (`kagent.dev/v1alpha1`) — a server kagent launches itself, over `stdio`, `sse`, or `streamableHttp`.

The course uses `RemoteMCPServer` pointing at the agentgateway data plane, because that keeps the tool runtime out of kagent's own process.

## How do you register the Ops Copilot tools?

The active resource in [`infra/kagent/toolserver.yaml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/infra/kagent/toolserver.yaml) is a `RemoteMCPServer` that points at the gateway's MCP endpoint:

```yaml
apiVersion: kagent.dev/v1alpha2
kind: RemoteMCPServer
metadata:
  name: agentops-tools
  namespace: agentops
spec:
  description: Ops Copilot read-only tools (incidents, service status, runbooks) over MCP.
  # Host/port/path must match the agentgateway Service (Ch. 6.5). /mcp is agentgateway's
  # streamable-HTTP MCP endpoint; use /sse with protocol SSE instead.
  url: http://agentgateway.agentops.svc.cluster.local:3000/mcp
  protocol: STREAMABLE_HTTP
  timeout: 30s
```

`description` and `url` are required; `protocol` defaults to `STREAMABLE_HTTP` and `timeout` to `30s`. The `url` is a Kubernetes Service DNS name — `agentgateway` in the `agentops` namespace on port 3000 — pointing at an in-cluster agentgateway. This course does **not** ship that Service (the local gateway config launches the MCP server over host-local `uv run`, which cannot run in-cluster), so applying this CR registers an endpoint that will not resolve until you deploy the gateway yourself. That is why it is left out of the build-to-deploy loop.

## When would you use a `ToolServer` instead?

When you want kagent to launch the MCP server itself rather than connect to a remote one — for example, running the Ops Copilot's MCP server directly over `stdio`. The commented alternative in the same file shows the shape:

```yaml
apiVersion: kagent.dev/v1alpha1
kind: ToolServer
metadata:
  name: agentops-tools-stdio
  namespace: agentops
spec:
  description: Ops Copilot MCP tools launched over stdio.
  config:
    stdio:
      command: uv
      args: [run, python, -m, agent.mcp_server]
```

A `ToolServer` config is **exactly one** of `stdio`, `sse`, or `streamableHttp`; `stdio` requires a `command`. The catch: the `stdio` command runs inside kagent's tool runtime, which must then have `uv` and the agent source available. The remote HTTP path avoids that coupling — the gateway owns the tool process, kagent just calls it — which is why the course prefers `RemoteMCPServer`.

## What is `MCPServer` and where does kmcp come in?

kagent bundles a subproject called **kmcp**, pulled in as a Helm dependency (`oci://ghcr.io/kagent-dev/kmcp/helm`) of the charts you installed in [6.2](./6.2. Platform Install.md). kmcp adds a third resource, **`MCPServer`**, and a controller that runs MCP servers as managed in-cluster workloads — kagent builds and hosts the server for you, rather than you pointing at an external URL. It is the natural next step when you want the platform to own the MCP server's lifecycle too; `RemoteMCPServer` is enough for this course because the gateway already hosts the tools.

## How does this connect to the gateway?

The `RemoteMCPServer` url — `http://agentgateway.agentops.svc.cluster.local:3000/mcp` — is the in-cluster address an **agentgateway** Service _would_ have, mirroring the MCP data plane you ran locally in [5.2. MCP Gateway](../5. Gateway/5.2. MCP Gateway.md). There, the gateway fronted the Ops Copilot's stdio MCP server and re-exposed its tools over streamable HTTP under `/mcp`. Running that same gateway in-cluster means co-locating it with the MCP source (for example as a sidecar of the agent pod, since the stdio launch needs `uv` and the agent code) — a deployment this course sketches but does not automate. [6.5](./6.5. Platform Gateway.md) explains the pattern and why it is left as an exercise.

!!! tip "One MCP endpoint, secured once"

    Routing tools through the gateway means every MCP call the deployed agent makes passes one policy point — auth, rate limiting, prompt guard — instead of scattering that concern across tool servers. The CRD only records _where_ the endpoint is; the gateway decides _what is allowed_.
