# 7.6. Governance

## What does governance mean for an agent?

Governance is how you stay accountable for what an autonomous system **does**. Everything else in this chapter observes the agent; governance _constrains_ it and _records_ it. Because an agent takes actions — it can restart a service or resolve an incident — the operative questions are: who approved this action, what policy allowed it, and can we prove afterwards exactly what happened? Governance is the set of controls that answer "yes" to all three.

For the Ops Copilot it rests on three pillars: **human-in-the-loop approvals** for risky actions, **policy** enforced in code, and an **audit log** that records every action for accountability.

!!! info "Governance is where AgentOps earns trust"

    You can ship an agent that is fast, cheap, and well-monitored and still not be allowed to run it — because no one can approve, constrain, or account for its actions. Governance is what lets an agent that _acts_ operate in a real organization: approvals bound what it may do, policy encodes the rules, and the audit log makes every action reviewable after the fact. It is the difference between a demo and a system you can put on call.

## How does human-in-the-loop approval work?

The agent may _propose_ a state-changing action, but a human must _approve_ it before it runs. The Ops Copilot's two mutating actions are **guarded** ([4.5. Guardrails](../4. Quality/4.5. Guardrails.md)): ADK pauses and requests confirmation before the function executes.

```python
# agents/python/src/agent/actions.py
# Guarded actions: ADK requests human approval before the function runs (HITL).
ACTION_TOOLS = [
    FunctionTool(func=restart_service, require_confirmation=True),
    FunctionTool(func=resolve_incident, require_confirmation=True),
]
```

The persona reinforces the same rule in plain language, so the model proposes rather than presumes:

```text
- Taking an action (restart_service, resolve_incident) changes state and needs human approval —
  propose it, and only call the tool when the engineer asks you to. Report the audit result.
```

Read-only tools (`list_incidents`, `get_incident`, `get_service_status`) run freely; only the actions that change the world require a human in the loop. That split — free to observe, gated to act — is the core governance boundary.

## What policy runs before an action?

A **before-tool callback** validates the action's inputs and can veto it before any state changes — a boundary check separate from the action's own logic. It fails fast on malformed inputs to the mutating actions:

```python
# agents/python/src/agent/guardrails.py
def validate_actions(tool: BaseTool, args: dict[str, Any], tool_context: ToolContext) -> dict[str, Any] | None:
    """Reject malformed inputs to mutating actions before they touch state."""
    del tool_context  # part of the ADK callback signature; unused here
    if tool.name not in _ACTION_TOOLS:
        return None
    if tool.name == "resolve_incident":
        incident_id = str(args.get("incident_id", ""))
        if not _INCIDENT_ID.match(incident_id):
            return {"error": f"Refusing to resolve {incident_id!r}: expected an id like INC-002."}
    if tool.name == "restart_service" and not str(args.get("name", "")).strip():
        return {"error": "Refusing to restart: no service name was provided."}
    return None
```

This is policy-as-code: the rules for what the agent may do live in a reviewed, tested function, not in a prompt the model can talk its way around. The rule lives in `validate_actions`. Two layers guard every action — the callback checks the inputs, and the human confirms the intent.

## How does the audit log record what the agent did?

Every mock action the agent performs is appended to an **append-only** audit trail in the shared dataset. The `audit_log` table is the schema:

```sql
-- agents/data/sql/schema.sql
-- Append-only trail of the mock actions the agent performs (restart, resolve, ...).
-- Guarded actions (Ch. 4.5) write here instead of touching real infrastructure.
CREATE TABLE audit_log (
    id     INTEGER PRIMARY KEY AUTOINCREMENT,
    ts     TEXT NOT NULL,                                  -- ISO-8601 UTC of the action
    actor  TEXT NOT NULL,                                  -- who/what performed it (e.g. "ops-copilot")
    action TEXT NOT NULL,                                  -- e.g. "restart_service", "resolve_incident"
    target TEXT NOT NULL,                                  -- service name or incident id
    detail TEXT NOT NULL
);
```

Each guarded action calls `append_audit`, which inserts one row and returns it, so the outcome is both persisted and reported back to the engineer:

```python
# agents/python/src/agent/data.py
def append_audit(actor: str, action: str, target: str, detail: str) -> dict[str, Any]:
    """Append one entry to the audit log and return it (used by mock actions, Ch. 4.5)."""
    timestamp = _utcnow()
    with _connect() as connection:
        cursor = connection.execute(
            "INSERT INTO audit_log (ts, actor, action, target, detail) VALUES (?, ?, ?, ?, ?)",
            (timestamp, actor, action, target, detail),
        )
        connection.commit()
        entry_id = cursor.lastrowid
    return {"id": entry_id, "ts": timestamp, "actor": actor, "action": action, "target": target, "detail": detail}
```

So `restart_service` flips the status _and_ records `actor="ops-copilot"`, `action="restart_service"`, the `target`, and a timestamp. The log answers **who did what, to what, and when** — the record accountability requires. Because it is append-only, it is a trustworthy history rather than mutable state.

## Why route mock actions through an audit log at all?

Because the audit log is the discipline you want _before_ the actions become real. In the course everything is mock and local — "restart" flips a database status; no infrastructure is touched. But the shape is exactly what production governance needs: a guarded action, a policy check, and an immutable record. When the same agent later drives real systems, only the action's body changes — the approval gate, the validation, and the audit trail are already in place.

The log is also an **observability** artifact. It complements the [trace](./7.1. Tracing.md) (which explains the reasoning behind an action) with a durable, queryable record of the action itself — the source of truth when someone asks "did the agent resolve INC-002, and who approved it?"

## How does governance relate to safety and compliance?

Governance is where safety and compliance become concrete and demonstrable:

- **Safety** — the HITL gate and input validation ([4.5. Guardrails](../4. Quality/4.5. Guardrails.md)) keep the agent from taking harmful or malformed actions, and pair with the injection defenses in [4.6. Security](../4. Quality/4.6. Security.md).
- **Compliance** — the audit log is evidence: an append-only record of every action, its actor, and its time, which is exactly what an audit or incident review needs.
- **Accountability** — approvals plus the trail mean every state change traces back to a human decision and a logged event, never to an unattributable autonomous act.

Together with the rest of Chapter 7 — reproducibility, tracing, monitoring, cost, feedback, and online evaluation — governance completes the picture: an agent you can not only see and afford, but also **trust to act**, and answer for when it does.
