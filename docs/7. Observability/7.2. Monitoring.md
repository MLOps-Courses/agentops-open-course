# 7.2. Monitoring

## Tracing versus monitoring — what is the difference?

A **trace** ([7.1](./7.1. Tracing.md)) explains _one_ run in detail: this request, these spans, this failure. **Monitoring** aggregates _many_ runs into a few numbers you watch continuously: how fast, how reliable, how expensive the agent is across all traffic. Tracing is for debugging a specific incident; monitoring is for noticing there _is_ an incident.

They share a pipeline. The same OTLP endpoint that carries spans also carries metrics, so once [7.1](./7.1. Tracing.md) is wired, your monitoring backend is already receiving data.

## Which metrics matter for an agent?

Four families cover most of what you need to know about a tool-using agent:

1. **Latency** — end-to-end response time per request, and per model call. Agents loop, so one answer can be several model calls; watch both the total and the per-step time.
1. **Error rate** — the fraction of runs that fail outright (model error, timeout, unhandled exception). This is the agent's availability.
1. **Tool failures** — how often tools return an error or no data. A rising tool-failure rate often means a dependency broke, not the model.
1. **Token throughput** — input and output tokens per request and per minute. This is both a load signal and the raw input to [cost](./7.3. Costs.md).

These map onto the quality metrics defined in [4.3. Metrics](../4. Quality/4.3. Metrics.md) — task success, tool accuracy, latency, cost — but measured continuously against live traffic instead of a fixed eval set.

## Where do these numbers come from?

From the spans ADK already emits. Because every model call, tool call, and workflow node is a timed span ([7.1](./7.1. Tracing.md)), your backend derives the metrics without extra instrumentation:

- **Latency** = span duration (the root span for end-to-end, the model spans for per-call).
- **Error rate** = fraction of spans marked with an error status.
- **Tool failures** = tool spans whose result is an `error` — and the Ops Copilot's tools return a structured `{"error": ...}` on bad input, so failures are legible in the trace rather than hidden in a stack trace.
- **Token throughput** = the token-count attributes carried on the model spans (see [7.3. Costs](./7.3. Costs.md)).

A collector (OpenTelemetry Collector, Grafana Tempo + Prometheus, or a hosted OTLP backend) turns these spans into dashboards and time series.

## What is an SLO for an agent, and how do I set one?

An **SLO** (Service Level Objective) is a target you commit to for an **SLI** (Service Level Indicator) — a measured metric — over a window. For the Ops Copilot, reasonable starting SLOs are:

| SLI                    | Example SLO                                   |
| ---------------------- | --------------------------------------------- |
| Availability           | ≥ 99% of requests succeed over 30 days        |
| Latency                | 95% of requests answer in under 8 seconds     |
| Tool success           | ≥ 98% of tool calls return data, not an error |
| Task success (offline) | 100% tool-trajectory pass on the eval set     |

Set targets from your baseline, not aspiration: measure a week of real traffic, then commit to a number you already meet most of the time. The gap between the SLO and 100% is your **error budget** — the room you have to ship changes and take risk before you must slow down and stabilize.

!!! tip "Agents need behavioral SLOs, not just uptime"

    A model endpoint is healthy if it returns 200s quickly. An agent can return 200s quickly and still be _wrong_ — calling the wrong tool, hallucinating an incident. So pair the infra SLIs above with a **quality** SLI sampled from live traffic ([7.5. Online Evaluation](./7.5. Online Evaluation.md)). "Fast and available" is necessary but not sufficient for an agent.

## What should I actually alert on?

Alert on **symptoms users feel**, not on every twitch. A good starting set:

1. **Error-budget burn** — page when the error or latency SLO is burning budget fast enough to exhaust it early. This catches real degradation without paging on a single slow request.
1. **Tool-failure spikes** — a sudden jump in tool errors usually means a dependency (the database, an MCP server, the gateway) is down. This is often your earliest, clearest signal.
1. **Latency regressions** — the p95 crossing its threshold, sustained over a few minutes, not a one-off.
1. **Cost anomalies** — token throughput far above baseline, which can mean a prompt-injection loop or a runaway retry. Covered in [7.3. Costs](./7.3. Costs.md).

Route these by severity: page a human for budget-burning outages, and send the slow-moving signals (a creeping tool-error rate, a cost drift) to a channel for review. Avoid alerting on raw counts that scale with traffic — alert on **rates and ratios** so the thresholds hold as load changes.

## How does monitoring connect to the rest of the loop?

Monitoring is where the [AgentOps loop](../0. Overview/0.2. AgentOps.md) detects that something needs attention, then hands off:

- A **tool-failure** alert → check the [trace](./7.1. Tracing.md), fix the tool or dependency.
- A **cost** anomaly → dig into [7.3. Costs](./7.3. Costs.md).
- A **quality** dip → sample traffic in [7.5. Online Evaluation](./7.5. Online Evaluation.md), and feed corrections back through [7.4. Feedback](./7.4. Feedback.md).

Monitoring tells you _that_ something changed; the rest of the chapter tells you _what_ and _what to do_.
