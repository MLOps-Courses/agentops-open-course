# 7.4. Feedback

## Why capture user feedback?

Because your users see failures your metrics miss. Monitoring ([7.2](./7.2. Monitoring.md)) tells you the agent was fast and returned a 200; it cannot tell you the answer was subtly wrong, cited the wrong runbook, or missed the point. The on-call engineer using the Ops Copilot _can_. Feedback is the channel that turns that human judgment into a signal you can act on — the ground truth that closes the [AgentOps loop](../0. Overview/0.2. AgentOps.md).

Without it, you are optimizing latency and cost while flying blind on the one thing that matters most: whether the agent is actually helping.

## What kinds of feedback should I collect?

Two kinds, cheap and rich:

1. **Explicit feedback** — the user tells you directly. A **thumbs up/down** on each answer, an optional comment, and — most valuable — a **correction** ("no, INC-002 is on `checkout`, not `inventory`"). Corrections are gold because they contain the right answer, not just a complaint.
1. **Implicit feedback** — the user's behavior reveals satisfaction. Did they accept the proposed action or reject it? Rephrase the same question three times? Abandon the session? These signals are noisier but free — you already have them in the trace.

Start with the thumbs and a comment box; add correction capture next, because a correction is a ready-made eval case.

## How do I capture it without losing context?

Attach feedback to the **trace**, not to a standalone form. Every answer already has a trace id ([7.1](./7.1. Tracing.md)); record the thumbs, comment, or correction against that id. That way each piece of feedback carries the full context for free — the exact prompt, the model, the tools called, and the response the user judged.

`adk web`'s trace view already links a run to its spans; a production UI adds a feedback control next to the answer and stores `{trace_id, verdict, comment}` in a small table. Keep the capture minimal — one click plus optional text — or users won't use it.

!!! note "Feedback is data, and data is governed"

    A thumbs-down often comes with a user's own words, which may contain sensitive detail. Treat the feedback store like any other user data: mind retention, access, and PII, and never let a raw correction flow straight into a public eval set unreviewed. Governance ([7.6](./7.6. Governance.md)) applies to what you collect, not just to what the agent does.

## Where does feedback go once I have it?

Feedback is only useful if it changes the agent. Route each kind to the artifact that fixes it:

1. **Into evals** — turn a correction into a new case in the fixed eval set ([4.4. Evaluations](../4. Quality/4.4. Evaluations.md)). A recurring thumbs-down on "what's the status of X" becomes a recorded tool-trajectory check, so the fix is verified and stays fixed. This is the single most valuable route: feedback becomes a regression test.
1. **Into skills and the prompt** — if the agent knows the answer but formats or reasons poorly, tighten a [skill](../3. Capabilities/3.2. Skills.md) or the persona's operating rules ([7.0](./7.0. Reproducibility.md)). Because the prompt is versioned code, that change is a reviewable diff.
1. **Into runbooks and memory** — if the agent is right about the data but the _guidance_ is wrong, the runbook is stale. Fix the `runbooks/*.md` the agent retrieves through [memory](../3. Capabilities/3.4. Memory.md); the correction improves every future incident of that type.

## How does this connect to online evaluation and monitoring?

Feedback and online evaluation ([7.5](./7.5. Online Evaluation.md)) are two ways of scoring live traffic: one asks the **user**, the other asks an **automated judge**. They reinforce each other — a spike in thumbs-down flags exactly which traffic to sample and evaluate, and an online-eval regression tells you where to solicit feedback.

The flow is a cycle: monitoring ([7.2](./7.2. Monitoring.md)) surfaces a dip, feedback explains it in the user's words, and the correction lands as a new eval case that guards against the regression forever. That is the observability loop feeding the build loop.
