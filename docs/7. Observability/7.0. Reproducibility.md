# 7.0. Reproducibility

## Why does reproducibility come first in observability?

Because you cannot debug, evaluate, or trust what you cannot reproduce. An agent is a non-deterministic system: the same prompt can yield different tool calls run to run. If _everything else_ drifts too — the model version, the persona, the dataset, the library versions — you can never tell whether a behavior changed because of your edit or because the ground moved under you.

Reproducibility is the control variable. Pin the version, the model, the prompt, and the data, and any change in behavior is attributable to the one thing you actually changed. That is what makes tracing ([7.1](./7.1. Tracing.md)), evaluation ([4.4](../4. Quality/4.4. Evaluations.md)), and online evaluation ([7.5](./7.5. Online Evaluation.md)) meaningful rather than noise.

## What has to be pinned to reproduce an agent run?

Four things move an agent's behavior. Pin all four:

1. **The model** — an explicit model id, not a provider default.
1. **The prompt** — the persona and operating rules, versioned as code.
1. **The data** — the dataset the agent reads and is evaluated against.
1. **The tools** — the exact library and toolchain versions that run it.

The Ops Copilot pins each of these on purpose. The rest of this page walks through how.

## How do we pin the model?

The model id is set **explicitly** and never left to a framework default, because ADK defaults churn between releases. It lives in one typed setting with a safe fallback and is overridable through `AGENT_MODEL`:

```python
# agents/python/src/agent/config.py
# Set an explicit model — ADK defaults churn. Override with AGENT_MODEL.
model: str = "gemini-3.5-flash"
```

The agent reads `AGENT_MODEL` from the `.env`. See [1.4. Providers](../1. Setup/1.4. Providers.md) for how the setting is loaded.

!!! tip "Pin the exact model, not a moving alias"

    `gemini-3.5-flash` is chosen deliberately. Aliases like a bare `flash` or a provider default can be repointed to a new snapshot without notice, which silently changes behavior. When you need byte-for-byte stability, pin the most specific model snapshot your provider offers and record it alongside the run.

## How do we pin the prompt?

The persona and operating rules are **code, not a runtime input**. They live in the agent module as a single constant, version-controlled and reviewed like any other source. Nothing prepends or mutates them at request time, so the instruction that shaped a trace is exactly the one in git at that commit:

```python
# agents/python/src/agent/agent.py
# The persona and operating rules. Kept explicit so behavior is reproducible and evaluable.
INSTRUCTION = """\
You are the Ops Copilot, an on-call assistant for a fictional online platform.
You help engineers triage and resolve incidents quickly and safely.

Operating rules:
- Always ground your answers in the tools. Never invent incidents, services, or statuses.
...
"""
```

The persona is carried as a versioned constant. Because the prompt is a string in source, a prompt change is a diff in a pull request — you can see it, review it, and roll it back.

## How do we pin the data?

The Ops Copilot runs against a **100% local, bundled dataset** — no external services, no live APIs. The `incidents.db` SQLite database is generated deterministically from committed SQL and shipped in-repo:

```bash
# agents/data — rebuild the database from the committed schema + seed
mise run build   # rm -f incidents.db && sqlite3 incidents.db < sql/schema.sql && sqlite3 incidents.db < sql/seed.sql
```

The schema and a fixed `seed.sql` are the single source of truth, so anyone who rebuilds gets the same rows. On top of that seed data sit **fixed eval sets** — recorded cases with a frozen prompt, expected tool trajectory, and reference response:

```json
// agents/python/evals/ops.evalset.json
{
  "eval_set_id": "ops_copilot_core",
  "name": "Ops Copilot core behaviors",
  "description": "Deterministic tool-trajectory checks over the bundled dataset (Ch. 4.4)."
}
```

A deterministic dataset plus a deterministic eval set is what lets [4.4. Evaluations](../4. Quality/4.4. Evaluations.md) gate on an exact tool trajectory (`tool_trajectory_avg_score: 1.0`) rather than fuzzy text overlap.

## How do we pin the tools and toolchain?

Every tool the course uses is version-locked so the environment rebuilds identically:

- **`mise.lock`** — pins the exact versions of the tools `mise` installs (uv, sqlite, dprint, …).
- **`uv.lock`** — pins the full Python dependency graph, including `google-adk`; `mise run check` enforces `uv lock --check`.

Because ADK ships fast, breaking releases, pinning is not optional. The lockfiles are the reason a run recorded today can be reproduced next month: the same code, the same model id, the same data, and the same library versions.

!!! warning "Model output is not bit-for-bit deterministic"

    Pinning removes every source of drift you control — but the model provider still runs the inference. Even at temperature 0, hosted models can vary slightly across snapshots and load. Reproducibility here means **the inputs are frozen and attributable**, not that the tokens are identical every call. That is why the course evaluates on **tool trajectory** (deterministic and meaningful) rather than exact wording — see [4.4. Evaluations](../4. Quality/4.4. Evaluations.md).

## How do I version the prompt and compare runs over time?

The prompt is already in git, which is the source of truth. But when you are iterating on the persona you also want to _compare_ versions and tie each one to the eval scores it produced — a job for a **prompt registry** and an **experiment tracker**. **[MLflow](https://mlflow.org/)** (Apache-2.0, fully self-hosted, no account) gives you both, backed by a local file — no server to run:

```python
# agents/python/evals/mlflow_eval.py — register the persona, then evaluate against it
import mlflow.genai

mlflow.set_tracking_uri("sqlite:///mlflow.db")  # local, zero-server; the Prompt Registry needs a SQL store
mlflow.genai.register_prompt(name="ops-copilot-instruction", template=INSTRUCTION)
```

Every `mise run eval:mlflow` ([4.4](../4. Quality/4.4. Evaluations.md)) registers the current instruction as a new **prompt version** and logs the run's scores against it, so `uv run mlflow ui --backend-store-uri sqlite:///evals/mlflow.db` (from `agents/python/`) shows the persona history and the metrics each version scored — the run-over-run view git alone does not give you. It is the same single-variable discipline, made queryable: change the prompt, re-run, and read which version scored better.

## What does "deterministic replay" actually give me?

A trustworthy A/B. When something regresses in production, you:

1. Read the trace ([7.1](./7.1. Tracing.md)) to recover the exact input, model, and commit.
1. Check out that commit — the pinned model, prompt, data, and lockfiles come with it.
1. Re-run the fixed eval set to reproduce the behavior locally, offline.
1. Change **one** variable — the prompt, a tool, the model id — and re-run.

Any difference in the tool trajectory is now caused by your one change, not by hidden drift. That single-variable discipline is the foundation the rest of Chapter 7 builds on: trace it ([7.1](./7.1. Tracing.md)), monitor it ([7.2](./7.2. Monitoring.md)), cost it ([7.3](./7.3. Costs.md)), and evaluate it live ([7.5](./7.5. Online Evaluation.md)).
