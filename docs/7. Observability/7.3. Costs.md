# 7.3. Costs

## Why is cost a first-class concern for agents?

Because an agent spends money every time it thinks. A stateless model call costs one request; an agent **loops** — it reasons, calls a tool, reads the result, reasons again — so a single user question can be several model calls, each billed per token. Costs compound with autonomy. Left unwatched, a retry storm or a prompt-injection loop can turn a cheap task into an expensive one without a single error in your logs.

Cost is therefore a metric you monitor ([7.2](./7.2. Monitoring.md)) and a constraint you design for, not a surprise you read on an invoice.

## How do I measure what a run costs?

From tokens. Providers bill per input and output token, so **token counts are the unit of cost**. You already have them: the model spans ADK emits ([7.1](./7.1. Tracing.md)) carry token-count attributes — prompt tokens in, completion tokens out — for every model call in the run.

To get **cost per task**, sum the tokens across all model spans in a single trace and multiply by the model's per-token price:

1. Group model spans by their root trace (one trace = one user task).
1. Sum input and output tokens across those spans.
1. Multiply by the input and output rates for `gemini-3.5-flash`.

Because the trace already ties every model call to one task, cost-per-task falls out of the same pipeline as latency and throughput — no separate accounting system.

## What is the cheap-model default, and why does it matter?

The Ops Copilot defaults to **`gemini-3.5-flash`** — a fast, low-cost model — set explicitly in both tracks:

=== "Python"

    ```python
    # agents/python/src/agent/config.py
    # Set an explicit model — ADK defaults churn. Override with AGENT_MODEL.
    model: str = "gemini-3.5-flash"
    ```

=== "Go"

    ```go
    // agents/go/internal/config/config.go
    // DefaultModel is set explicitly because ADK model defaults churn.
    const DefaultModel = "gemini-3.5-flash"
    ```

Choosing a cheap, capable model as the baseline is the single biggest cost lever, because it multiplies across every loop step of every task. You reach for a larger model only when evals show the cheap one failing a specific case — not by default. And for **evaluation** workloads, which run the whole set repeatedly, prefer an even cheaper tier (a Flash-Lite class model) so the offline and online eval loops ([4.4](../4. Quality/4.4. Evaluations.md), [7.5](./7.5. Online Evaluation.md)) stay affordable to run often.

!!! tip "The cheapest token is the one you never send"

    Model choice is one lever; **prompt size** is the other. Every runbook you stuff into context and every needless tool round-trip is billed. The Ops Copilot keeps costs down by design: [skills](../3. Capabilities/3.2. Skills.md) load procedures on demand instead of pre-loading them, and [memory](../3. Capabilities/3.4. Memory.md) retrieves only the runbook a task needs. Fewer, tighter calls beat a cheaper model.

## What budgets and KPIs should I track?

Turn cost into numbers you watch, the same way you set SLOs for latency ([7.2](./7.2. Monitoring.md)):

- **Cost per task** — the headline KPI: average and p95 spend to answer one question. A rising p95 means some tasks are looping too much.
- **Tokens per task** — the provider-independent version of the same signal; watch it so a price change and a behavior change don't blur together.
- **Cost per successful task** — cost divided by task-success rate ([4.3. Metrics](../4. Quality/4.3. Metrics.md)). Cheap answers that are wrong are not cheap.
- **Daily / monthly budget** — a spend ceiling with an alert well before it. Pair it with a rate limit at the [gateway](../5. Gateway/5.5. Gateway Security.md) so a runaway agent hits a hard stop, not just a dashboard.

## What should trigger a cost alert?

Alert on **anomalies and budget burn**, not on absolute spend that scales with traffic:

1. **Token throughput far above baseline** — a spike usually means retries, a loop, or an injection dragging the agent into extra calls.
1. **Cost-per-task regression** — the p95 climbing after a prompt or tool change you shipped; catch it before it compounds across all traffic.
1. **Budget pace** — spending fast enough to blow the monthly budget early, the cost analogue of error-budget burn.

Route a budget-threatening spike to a human, and a slow cost drift to a review channel — the same severity split as [7.2. Monitoring](./7.2. Monitoring.md).

## Where does cost control actually live?

In three places, in order of leverage:

1. **The agent** — the cheap default model, on-demand skills, and retrieval keep per-task cost low at the source.
1. **The gateway** ([Chapter 5](../5. Gateway/)) — rate limits and per-consumer quotas cap spend and stop runaway loops centrally, across both tracks.
1. **The dashboard** ([7.2](./7.2. Monitoring.md)) — budgets and alerts make the spend visible so you act before the invoice does.

Cost, latency, and quality trade off against each other; keeping all three on the same dashboard is what lets you tune one without silently wrecking another.
