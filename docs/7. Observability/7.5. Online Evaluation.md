# 7.5. Online Evaluation

## Offline versus online evaluation — what is the difference?

**Offline evaluation** ([4.4. Evaluations](../4. Quality/4.4. Evaluations.md)) runs a **fixed** set of recorded cases against a known dataset before you ship. It is deterministic, repeatable, and gates your releases — but it only measures the situations you thought to record.

**Online evaluation** scores the agent on **real traffic**, in production, on inputs you never anticipated. It answers the question offline evals cannot: is the agent still good on the messy, evolving questions users actually ask? Offline evals prove the agent is ready to ship; online evals prove it is still working after it shipped.

You need both. Offline is your regression gate; online is your smoke detector.

## Why isn't the offline eval set enough?

Because production drifts away from your fixtures in ways a frozen set cannot see:

1. **Input drift** — users ask new kinds of questions; new services and incident types appear that no recorded case covers.
1. **Model drift** — the hosted model updates underneath you. Even with the model id [pinned](./7.0. Reproducibility.md), a provider can repoint a snapshot, shifting behavior on inputs your eval set never exercised.
1. **Data drift** — the world the agent reasons over changes: runbooks are edited, the incident mix shifts.

The offline set is a fixed photograph; production is a moving scene. Online evaluation is how you notice the scene has moved.

## How do I evaluate live traffic without evaluating everything?

**Sample** it. Scoring every request is expensive and unnecessary; a representative slice is enough to spot trends. A practical pipeline:

1. **Sample** a small percentage of live traces ([7.1](./7.1. Tracing.md)) — plus every trace that drew a thumbs-down ([7.4. Feedback](./7.4. Feedback.md)), because those are the ones most worth examining.
1. **Score** each sampled run. Where there is a known answer, reuse the offline **tool-trajectory** check — did the agent call the right tools? Where there is not, use an **LLM-as-judge** to rate groundedness and helpfulness against the trace.
1. **Aggregate** the scores into a quality metric over time and watch it on the same dashboard as latency and cost ([7.2. Monitoring](./7.2. Monitoring.md)).

!!! tip "Judge with a cheap model"

    An LLM judge runs on sampled traffic continuously, so its cost adds up. Use a cheap tier — a Flash-Lite class model — as the judge, the same economy the course applies to eval workloads ([7.3. Costs](./7.3. Costs.md)). Calibrate it against human labels on a handful of cases first, then let it grade the stream; a mis-calibrated judge is worse than no judge.

## How do I detect drift?

Drift is a **trend in the aggregate**, not a single bad answer. Watch the online quality score over time and alert when it degrades — the behavioral SLI that complements the infra SLIs of [7.2. Monitoring](./7.2. Monitoring.md):

- A slow decline in groundedness → the model or the runbooks are drifting.
- A rise in a specific tool's failure rate → a dependency or a schema changed.
- A cluster of thumbs-down on one topic → an input class the agent was never taught.

When the aggregate dips, drop back to individual traces to find the cause — monitoring points, tracing explains.

## What do I do when online evals flag a regression?

Close the loop back to offline. The whole point is that a production failure becomes a permanent guardrail:

1. Take the failing live trace and **reproduce** it locally against the pinned model, prompt, and dataset ([7.0. Reproducibility](./7.0. Reproducibility.md)).
1. **Record** it as a new case in the fixed eval set ([4.4. Evaluations](../4. Quality/4.4. Evaluations.md)) — now it is a deterministic regression test.
1. **Fix** the cause: a prompt tweak, a sharper [skill](../3. Capabilities/3.2. Skills.md), an updated [runbook](../3. Capabilities/3.4. Memory.md), or a tighter [guardrail](../4. Quality/4.5. Guardrails.md).
1. **Verify** the new case passes offline, then ship.

Online evaluation discovers the problem; offline evaluation makes sure it never comes back. Together with [feedback](./7.4. Feedback.md), that is the observability half of the AgentOps loop feeding the build half.
