# 7.1. Tracing

## Why trace an agent at all?

Because an agent run is a chain of hidden decisions. A single user question can trigger several model calls, each choosing tools, each feeding results back into the loop. When the answer is wrong, slow, or expensive, a log line does not tell you _which_ step went wrong. A **trace** does: it records the whole run as a tree of timed **spans** — the model calls, the tool calls, and the workflow nodes — so you can see exactly what the agent did, in what order, and how long each step took.

Tracing is the raw signal the rest of the chapter refines into monitoring ([7.2](./7.2. Monitoring.md)), cost ([7.3](./7.3. Costs.md)), and online evaluation ([7.5](./7.5. Online Evaluation.md)).

## Do I have to instrument my code?

No. ADK **auto-instruments** the agent for you. Out of the box it emits OpenTelemetry spans for every model call, every tool call, and every workflow node — no decorators, no manual span creation in your tool functions. You write the agent exactly as in [Chapter 3](../3. Capabilities/); the spans appear for free.

What you _do_ configure is where those spans go. By default they are collected but not exported anywhere; to send them to a collector you set the standard `OTEL_*` environment variables.

## How do I see traces with zero setup?

Run the developer web UI. `adk web` (Python) and the Go launcher's web mode both show the same trace tree in a built-in view — every model and tool span for the run you just triggered, with no exporter to configure:

=== "Python"

    ```bash
    # agents/python
    mise run web        # uv run adk web src → opens the ADK dev UI with a Trace view
    ```

=== "Go"

    ```bash
    # agents/go — the full launcher exposes the same web UI
    mise run run -- web  # go run ./cmd/agent web
    ```

This is the fastest inner loop: ask the Ops Copilot a question, open the trace, and confirm it called `get_service_status` before recommending a runbook. See [2.6. Dev Loop](../2. Agents/2.6. Dev Loop.md) for the dev UI itself.

## How do I export traces to a collector?

Point the agent at an OTLP endpoint with environment variables. The `.env.example` ships the two you need (commented out); uncomment them in your `.env`:

```bash
# .env — Observability (Ch. 7): point at an OTLP collector to export traces/metrics/logs.
# ADK auto-instruments model and tool calls; `adk web` also shows traces in its UI.
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318
OTEL_SERVICE_NAME=ops-copilot
```

`OTEL_EXPORTER_OTLP_ENDPOINT` is the OTLP receiver (a collector, Jaeger, Grafana Tempo, or the [gateway's](../5. Gateway/5.6. Gateway Observability.md) OTLP endpoint). `OTEL_SERVICE_NAME` labels the traces so the Ops Copilot is distinguishable from other services in the backend. These are the vendor-neutral OpenTelemetry variables — nothing ADK-specific — so any OTLP-compatible backend works.

## How is exporting wired in each track?

Both tracks read the **same** `OTEL_*` variables; the mechanics differ slightly.

=== "Python"

    A tiny helper enables the OTLP exporters from the environment. It is a no-op when no endpoint is set, so importing it is always safe. Call it once at process start for programmatic runs; `adk web` / `adk run` set it up for you.

    ```python
    # agents/python/src/agent/telemetry.py
    """OpenTelemetry setup for the Ops Copilot (Chapter 7.1).

    ADK already emits OTel spans for every model call, tool call, and workflow node. This helper
    wires the *exporters* so those spans reach a collector (Jaeger, Grafana Tempo, the gateway's
    OTLP endpoint, …) from the standard ``OTEL_*`` environment variables. It is a no-op unless an
    endpoint is configured, so importing it is always safe. ``adk web`` shows the same traces in
    its built-in trace view without any setup.
    """

    from __future__ import annotations

    from google.adk.telemetry.setup import maybe_set_otel_providers


    def setup_telemetry() -> None:
        """Enable OTLP tracing/metrics/logs from ``OTEL_EXPORTER_OTLP_ENDPOINT`` (and friends).

        Call this once at process start for programmatic runs; ``adk web``/``adk run`` set it up
        for you. No endpoint configured → nothing is exported.
        """
        maybe_set_otel_providers()
    ```

=== "Go"

    The launcher initializes OpenTelemetry for you and reads the same `OTEL_EXPORTER_OTLP_ENDPOINT`, so setting the env var exports traces with **no code change**. For finer control, set `TelemetryOptions` on the launcher `Config` (the same `cfg` from [2.2](../2. Agents/2.2. First Agent (Go).md)):

    ```go
    // agents/go/cmd/agent/main.go — no telemetry code is required; this is the optional hook.
    cfg := &launcher.Config{
        AgentLoader:      adkagent.NewSingleLoader(a),
        TelemetryOptions: []telemetry.Option{ /* custom resource, sampler, exporter… */ },
    }
    ```

The pattern is symmetric: ADK produces the spans; the environment decides where they land. Nothing about your tools or agent code changes when you turn exporting on.

## How do I run a collector locally?

Run an all-in-one Jaeger, which speaks OTLP and gives you a trace UI:

```bash
# Jaeger with the OTLP/HTTP receiver on :4318 and the UI on :16686
docker run --rm --name jaeger \
  -p 16686:16686 -p 4318:4318 \
  jaegertracing/all-in-one:latest
```

Then set `OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318` in your `.env`, run the agent, and open the Jaeger UI at `http://localhost:16686` to see the Ops Copilot's spans. Swap Jaeger for the OpenTelemetry Collector or Grafana Tempo when you want to fan the same spans out to multiple backends — the agent side does not change.

!!! note "One pipeline for traces, metrics, and logs"

    The same OTLP endpoint carries traces, metrics, and logs. That is why the next pages build directly on this one: [7.2. Monitoring](./7.2. Monitoring.md) reads the metrics off the same pipeline, and [7.3. Costs](./7.3. Costs.md) reads token counts off the span attributes. Set the endpoint once and the whole chapter lights up.

Once traces flow, turn them into health signals with [7.2. Monitoring](./7.2. Monitoring.md).
