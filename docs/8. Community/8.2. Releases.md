# 8.2. Releases

## What is a release?

A release is a formal, versioned snapshot of a project at a stable point in its history. It bundles the code (and any artifacts — container images, binaries, the built docs) behind a single version number, together with a changelog that says what moved since the last one. For an agent, a release is a contract: pinning `v1.2.3` guarantees a specific set of tools, prompts, and behaviors — which matters a lot when the agent ecosystem underneath you churns.

## How do you version an agent project?

Use [Semantic Versioning (SemVer)](https://semver.org/): a `MAJOR.MINOR.PATCH` number where each part signals the kind of change.

1. **MAJOR** — incompatible, breaking changes (a removed tool, a changed agent interface).
1. **MINOR** — new, backward-compatible capability (a new tool or workflow).
1. **PATCH** — backward-compatible fixes (a prompt tweak, a bug fix).

This repository declares its version in [`pyproject.toml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/pyproject.toml) (`version = "0.1.0"`). A `0.x` version signals that things are still stabilizing — exactly right for a course whose stack is pre-1.0 — and gives you room to iterate before committing to `1.0.0`.

!!! tip "Pin the volatile pieces separately"

    SemVer versions *your* project; it does not tame your dependencies. The agent stack moves fast, so also pin the moving parts explicitly — `google-adk`, the agentgateway image tag, the kagent chart — as [AGENTS.md](https://github.com/MLOps-Courses/agentops-open-course/blob/main/AGENTS.md) requires. A reproducible release pins both your version and theirs.

## What goes in the changelog, and how is it generated?

The changelog is the human-readable story of a release. Rather than writing it by hand, generate it from your commit history with [git-cliff](https://git-cliff.org/), which reads [Conventional Commits](https://www.conventionalcommits.org/) and groups them into sections:

- `feat:` commits become **Features**.
- `fix:` commits become **Bug Fixes**.
- `docs:`, `chore:`, `refactor:` and the rest are grouped or filtered per your config.

Because contributions to this repository already use Conventional Commits (see [8.5. Contributions](./8.5. Contributions.md)), a changelog can be generated straight from the commit log — no manual bookkeeping, and no drift between what changed and what you claim changed. This is the recommended release flow below; the course itself is pre-1.0 and does not yet ship a `cliff.toml` or a `CHANGELOG.md`, so wiring `git-cliff` in is the first step when you cut a real release.

## How do you cut a release?

The flow is small and repeatable — a few steps you can run locally, then hand off to GitHub:

1. **Bump the version** in `pyproject.toml`, following SemVer for the kind of change.
1. **Generate the changelog** with `git-cliff`, writing or updating `CHANGELOG.md`.
1. **Commit** the bump and changelog (e.g., `chore: release v0.2.0`).
1. **Tag** the commit with a `v`-prefixed tag, `git tag v0.2.0`, and push the tag.
1. **Publish** a GitHub release from that tag — `gh release create v0.2.0 --notes-file CHANGELOG.md` — attaching the changelog notes (and any built artifacts).

Pushing to `main` also triggers the [`docs.yml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/.github/workflows/docs.yml) workflow, so the published documentation is rebuilt to match the release (see [8.4. Documentation](./8.4. Documentation.md)).

## Why v-prefixed tags?

The `v` prefix (`v0.2.0`, not `0.2.0`) is the long-standing convention across Git tooling. GitHub, git-cliff, and `gh release` all understand the same `v`-shaped tags, so one convention drives the whole release.
