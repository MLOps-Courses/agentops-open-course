# 8.3. Templates

## Why turn your agent into a template?

Once you have a working agent — configured, tested, containerized, deployable — most of that scaffolding is not specific to _this_ agent. A template captures the reusable skeleton so your next agent starts at the finish line of this course instead of a blank folder. You stop copy-pasting a past project and hand-editing names, and you stop drifting: every agent your team ships shares the same layout, task vocabulary, and quality gates.

## What makes a good agent template?

The reference agent already shows the shape worth reusing. A good template parameterizes the details that change per project and keeps the structure that does not:

- **Parameterize** the agent name, module/package path, default `model`, and provider settings — the handful of values that differ every time.
- **Keep** the self-contained project layout, the [`mise run`](../1. Setup/1.0. System.md) task vocabulary (`install`/`format`/`check`/`test`), the tests and evals folders, the `Dockerfile`/`ko` build, and the CI workflows.
- **Keep** an [`AGENTS.md`](./8.0. Repository.md) and `README.md` so every generated project is immediately legible to both agents and humans.

The `agents/python` and `agents/go` projects are effectively templates already: each is standalone, pins its versions, and exposes the same tasks. Turning either into a generator is mostly a matter of replacing the fixed names with variables.

## How do you build one?

Reach for a scaffolding tool rather than a shell script. The two common choices are:

- **[Cookiecutter](https://cookiecutter.readthedocs.io/)** — the classic Python project templater. You mark variables in filenames and file contents with `{{ cookiecutter.agent_name }}` and answer prompts at generation time.
- **[Copier](https://copier.readthedocs.io/)** — a modern alternative that can also _update_ projects when the template changes, which is valuable when your baseline evolves.

The recipe is the same for both: copy your working agent, replace the project-specific strings (name, module path, default model) with template variables, add a small config listing those variables, and generate a fresh project to confirm it builds and tests green.

## What OSS starters already exist?

You do not have to start from scratch. Two Google-backed, open-source options are worth knowing:

- **The `adk` CLI scaffolder (Python)** — the Python ADK ships a built-in generator. `adk create my_app` "creates a new app in the current folder with prepopulated agent template", and `--model` presets the root agent's model:

  ```bash
  adk create my_app --model gemini-3.5-flash
  ```

  This is a Python-CLI feature; the Go CLI (`adkgo`) currently offers deploy commands rather than a scaffolder, so Go users lean on Copier/cookiecutter or the agent-starter-pack below.

- **[Google's agent-starter-pack](https://github.com/GoogleCloudPlatform/agent-starter-pack)** — an open-source CLI that generates more complete, production-oriented agent projects (tests, deployment, observability wired in). Use it when you want a fuller opinionated baseline rather than a minimal scaffold.

!!! tip "Templates are a versioning discipline"

    A template is another thing to keep pinned and released (see [8.2. Releases](./8.2. Releases.md)). Version your template, and when the underlying stack moves — a new ADK, a new agentgateway image — bump the template so every project generated from it inherits the fix.

## When should you _not_ build a template?

Templates pay off across many similar agents; for a single one, they are overhead. Reach for a template when you are standardizing a fleet or onboarding a team — otherwise just fork the reference agent, rename it, and move on. Match the tool to the scale of the reuse.
