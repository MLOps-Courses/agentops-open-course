# 8.4. Documentation

## What builds this site?

The site you are reading is built by **[Zensical](https://zensical.org/)** — a fast, Rust-powered static-site engine from the Material for MkDocs team. It reads plain Markdown from `docs/` and renders the whole course. Zensical is the only build dependency of the root project, pinned in [`pyproject.toml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/pyproject.toml):

```toml
dependencies = [
  "zensical==0.0.46", # alpha: reads mkdocs.yml natively; pin exactly (== on an alpha) and re-verify on upgrade
]
```

The `0.0.x` version is deliberate: Zensical is still alpha, so the version is pinned and re-verified at build time.

## Why is the config still called mkdocs.yml?

Because Zensical **reads `mkdocs.yml` natively**. Rather than adopt a new `zensical.toml` format while the engine is alpha and its schema still moves, the project keeps a well-understood [`mkdocs.yml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/mkdocs.yml) — the same config format the sibling MLOps course uses. The file itself says so at the top:

```yaml
# Read natively by Zensical (https://zensical.org). We keep the config in mkdocs.yml
# while Zensical is alpha; migrate to zensical.toml once it reaches 1.0.
```

The theme, navigation features, and Markdown extensions are declared there:

```yaml
theme:
  name: material
  palette:
    scheme: default
    primary: indigo
  features:
    - content.code.copy
    - navigation.footer
    - navigation.indexes
    - navigation.tabs

markdown_extensions:
  - admonition
  - attr_list
  - md_in_html
  - pymdownx.details
  - pymdownx.highlight:
      anchor_linenums: true
  - pymdownx.inlinehilite
  - pymdownx.superfences: # custom fence for mermaid diagrams
```

Two of these carry the course's rendering: `pymdownx.superfences` renders the `mermaid` diagrams, and `admonition` gives the `!!!` callouts you see throughout.

## How do I run and build it locally?

Through the root `mise` tasks, which wrap the `zensical` CLI:

```bash
mise run serve   # uv run zensical serve → live-reload preview at http://localhost:8000/
mise run build   # uv run zensical build → static site into site/
```

`mise run serve` (alias `mise run s`) rebuilds on save while you write; `mise run build` (alias `b`) produces the deployable `site/`. That same `zensical build` is also invoked by `mise run check`, so a broken page fails the same gate contributors and CI run.

## How is the site published?

Automatically, on every push to `main`, by the [`docs.yml`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/.github/workflows/docs.yml) GitHub Actions workflow. It has two jobs:

1. **build** — installs the pinned tools with `mise`, runs `mise run deploy` (which is `uv run zensical build`; `uv run` auto-syncs from the lock), and uploads the resulting `site/` with `actions/upload-pages-artifact`.
1. **deploy** — publishes that artifact to **GitHub Pages** with `actions/deploy-pages`.

There is no manual publish step: merge to `main`, and the live site updates. The same `zensical build` you run locally is the one CI runs, so what you preview is what ships.

## How does the custom domain work?

The site is served from **`agentops-open-course.fmind.dev`**, not a `github.io` URL. That is set two ways, and both must agree:

- **[`docs/CNAME`](https://github.com/MLOps-Courses/agentops-open-course/blob/main/docs/CNAME)** contains the single line `agentops-open-course.fmind.dev`. Zensical copies files under `docs/` into `site/`, so the `CNAME` lands at the site root where GitHub Pages expects it.
- **DNS** for `fmind.dev` points that subdomain at GitHub Pages, and `site_url` in `mkdocs.yml` is set to the same `https://agentops-open-course.fmind.dev/` so generated links and metadata are correct.

This mirrors the sibling course at `mlops-coding-course.fmind.dev`, keeping the two courses consistent under one domain.
