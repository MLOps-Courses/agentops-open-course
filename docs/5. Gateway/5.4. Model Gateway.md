# 5.4. Model Gateway

## How do I reach a non-Gemini model without changing the agent?

Chapters 2–4 run the agent on **native Gemini** — the simplest, most idiomatic ADK setup. To reach any other provider (OpenAI, Anthropic, OpenRouter, or a local **Ollama**), you do **not** swap SDKs or add LiteLLM. You add an **`ai` backend** to the gateway and repoint the agent's model **base URL** at that route. The agent keeps its native `google-genai` / Gemini client; only a URL changes. One switch — just a URL.

This is the payoff of the provider decision in the course plan: native Gemini first, true vendor-neutrality via the gateway — no per-provider client library.

## What is an `ai` backend?

`ai` is one of agentgateway's backend types (alongside `mcp` and `host`). It proxies LLM calls: it accepts an inference request, applies any policies, translates the request into the target provider's API, and forwards it. The provider is declared inside the backend — `openAI`, `anthropic`, `gemini`, `vertex`, `bedrock`, `azure`, and others are supported. Two fields let you retarget a provider's default host:

1. **`hostOverride`** — send the request to a different host:port than the provider default.
1. **`pathOverride`** — send it to a different path than the provider default.

Ollama exposes an **OpenAI-compatible** API at `http://localhost:11434/v1`, so you use the `openAI` provider and override the host to point at Ollama.

## What does the Ollama backend look like?

The active `infra/agentgateway/config.yaml` keeps the 5.4 block commented, ready to add as its own listener/route:

```yaml
# --- Chapter 5.4 — Model Gateway (reach any provider, incl. local Ollama) -----
# An `ai` backend proxies LLM calls, so you switch providers by pointing the agent's
# model base URL at this route — no SDK change. Ollama exposes an OpenAI-compatible
# API at http://localhost:11434/v1, so use the openAI provider and override the host.
# The gateway assumes TLS to upstreams, so a plain-HTTP local Ollama needs the
# `backendTLS: { insecure: true }` opt-out below (drop both it and hostOverride to reach
# api.openai.com directly). Add as its own route under the listener:
#
#      - policies:
#          backendTLS:
#            insecure: true # plain-HTTP local backend: skip the default upstream TLS
#        backends:
#          - ai:
#              name: ollama
#              hostOverride: localhost:11434
#              pathOverride: /v1/chat/completions
#              provider:
#                openAI:
#                  model: llama3.2
```

To reach the real OpenAI instead, drop `hostOverride` (the `openAI` provider defaults to `api.openai.com`) and set the model you want. To swap to Anthropic or Gemini, change the `provider` block — the agent's base-URL setting never moves. This matches the upstream `ai-prompt-guard` example, whose backend is the same shape:

```yaml
backends:
  - ai:
      name: openai
      provider:
        openAI:
          model: gpt-4o-mini
```

!!! note "Plain-HTTP local backends"

    A local Ollama speaks plain HTTP, not TLS. The gateway assumes TLS to upstreams by default, so
    the route sets `policies.backendTLS.insecure: true` (shown above) for a local backend. Without
    it you chase a TLS handshake error against a plain-HTTP port.

## How does the native Gemini client target a custom base URL?

The whole approach hinges on the ADK's native model client honouring a **base-URL override**. It does — the underlying `google-genai` client resolves its base URL in this priority order (verified in the installed SDK, `google/genai/_base_url.py`):

1. An explicit `HttpOptions(base_url=…)` passed to the client.
1. A value set via the SDK's `set_default_base_urls(...)`.
1. An environment variable — **`GOOGLE_GEMINI_BASE_URL`** (direct Gemini API) or **`GOOGLE_VERTEX_BASE_URL`** (Vertex AI).

So the switch is a single environment variable pointing the client at the gateway's `ai` route instead of Google's endpoint — no code edit. You can also set it explicitly on the client, if you prefer wiring over environment:

```python
from google.adk.models import Gemini

# ADK's Gemini model takes a first-class base_url; pass a configured model object to
# Agent(model=...) instead of the bare string from agent.py. Everything else is unchanged.
model = Gemini(model=settings.model, base_url="http://localhost:3000")
```

Either way, the agent code stays on its native client; the gateway does the provider translation.

!!! warning "Validate the wiring end-to-end"

    The gateway normalises inference request formats between providers, but the exact ingress
    format the native client sends (Gemini vs OpenAI-compatible) is the one thing to confirm
    against a running backend. Bring up a local Ollama, point `GOOGLE_GEMINI_BASE_URL` at the
    gateway's `ai` route, and run one completion before relying on it in an evaluation. Pin the
    model (`AGENT_MODEL`) explicitly — defaults churn.

## What changes in the config?

The agent already reads the model from the environment, so switching provider is a config change, not a code change: `agents/python/src/agent/config.py` reads `AGENT_MODEL` (default `gemini-3.5-flash`). Set `AGENT_MODEL` to the gateway-served model and export `GOOGLE_GEMINI_BASE_URL=http://localhost:3000` (or `GOOGLE_VERTEX_BASE_URL` on the Vertex path).

Point the agent at the gateway and you get provider-neutral behaviour — which is exactly what the [4.4. Evaluations](../4. Quality/4.4. Evaluations.md) suite needs to compare providers fairly.
