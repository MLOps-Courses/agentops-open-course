# 5.6. Gateway Observability

## Why observe at the gateway?

The gateway sees **every** MCP tool call, A2A hop, and LLM request the agent makes. That makes it the one place where you can measure the agent's outward behaviour without instrumenting the agent code itself. You get three signals for free or nearly free:

1. **Metrics** — request and tool-call counters, enabled by default.
1. **Tracing** — OpenTelemetry spans exported to any OTLP backend.
1. **The admin UI** — a live view of the running configuration at `localhost:15000`.

This is the front door to Chapter 7; the gateway's traces and metrics feed the broader observability story in [7.1. Tracing](../7. Observability/7.1. Tracing.md) and [7.2. Monitoring](../7. Observability/7.2. Monitoring.md).

## What metrics does the gateway expose?

Metrics are **on by default** — no configuration needed. The gateway exposes a Prometheus-style endpoint you can scrape. The upstream `telemetry` example shows the shape: per-tool call counters, list-call counters, and per-status request totals:

```text
tool_calls_total{server="everything",name="echo"} 1
tool_calls_total{server="everything",name="add"} 1
list_calls_total{resource_type="tool"} 3
agentgateway_requests_total{gateway="bind/3000",method="POST",status="200"} 9
agentgateway_requests_total{gateway="bind/3000",method="GET",status="200"} 1
```

For the Ops Copilot you would see `tool_calls_total` broken down by the `ops-copilot` server and each tool name (`list_incidents`, `get_service_status`, …) — exactly the tool-accuracy signal [4.3. Metrics](../4. Quality/4.3. Metrics.md) defines. Point Prometheus at the metrics endpoint to collect them.

## How do I turn on tracing?

Add a `tracing` policy that names an OTLP endpoint. The gateway then emits a span per request. The upstream `telemetry` example config sets it under `frontendPolicies`:

```yaml
frontendPolicies:
  tracing:
    host: localhost:4317
    randomSampling: true
binds:
  - port: 3000
    listeners:
      - routes:
          - backends:
              - mcp:
                  targets:
                    - name: everything
                      stdio:
                        cmd: npx
                        args: ["@modelcontextprotocol/server-everything"]
```

`host` is the OTLP collector address; `randomSampling: true` samples traffic. Bring up any OTLP backend — the example ships a `docker compose` with an OpenTelemetry Collector and Jaeger — then open the Jaeger UI to see spans for each MCP call. Because the spans are standard OpenTelemetry, they land in the same tracing pipeline as the agent's own spans in Chapter 7.

!!! note "Tracing config has moved between versions"

    Older examples set tracing under a top-level `config.tracing.otlpEndpoint`; v1.3.1 examples use
    the `frontendPolicies.tracing` form shown above. Match the form your pinned version accepts —
    this is one of the schema areas that still moves, so verify against the running binary.

## Can I enrich traces and logs with the authenticated user?

Yes — that is where security (5.5) and observability meet. Once a route authenticates callers, you can surface JWT claims in access logs. The upstream `oidc` example adds the user identity to every log line:

```yaml
frontendPolicies:
  accessLog:
    add:
      user.id: jwt.sub
      user.email: jwt.email
```

Now each request is attributable to a user, and A2A requests carry protocol-specific fields (like `a2a.method`) in the log — so "who called which tool, when, and how long it took" is answerable from the gateway alone.

## What is the admin UI for?

The gateway serves an admin UI on **`localhost:15000`** (see [5.1. Gateway Setup](./5.1. Gateway Setup.md)). Use it to:

1. Confirm the config loaded — inspect the live binds, listeners, routes, and backends.
1. See which policies are attached to each route.
1. Debug a misconfiguration without re-reading the YAML.

It is the quickest sanity check that the gateway is running the config you think it is, and a handy companion to the metrics and traces when something looks off.

## How does this carry into Kubernetes?

The same metrics and traces flow when the gateway runs as kagent's in-cluster data plane ([6.5. Platform Gateway](../6. Platform/6.5. Platform Gateway.md)). You scrape the metrics endpoint with the cluster's Prometheus and export traces to the cluster's OTLP collector — no change to the agent. Chapter 7 assembles these into dashboards, cost tracking, and SLOs.
