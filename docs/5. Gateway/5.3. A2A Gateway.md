# 5.3. A2A Gateway

## What is A2A traffic through the gateway?

[A2A (Agent2Agent)](../3. Capabilities/3.6. A2A.md) lets one agent call another over HTTP. When the Ops Copilot's triage agent delegates to a diagnosis agent, that call is A2A traffic. Putting the gateway in the middle gives you the same benefits as fronting MCP: one place for auth, rate limits, and tracing on agent-to-agent hops — and a stable, gateway-owned URL so callers cannot bypass the proxy.

## Is A2A a backend type or a policy?

**A2A is a route _policy_, not a backend.** This is the single most important thing to get right. You do not write `a2a:` under `backends:`. Instead you:

1. Add a route whose backend is a plain **`host`** backend pointing at the agent's own A2A server.
1. Attach the **`a2a: {}`** policy to that route to mark the traffic as A2A.

The gateway then processes the traffic as A2A: it recognises the protocol and rewrites the agent card so downstream callers keep coming back through the gateway.

## What does the config look like?

The active `infra/agentgateway/config.yaml` keeps the A2A block commented, ready to add as a second route under the existing listener:

```yaml
# --- Chapter 5.3 — Agent Gateway (A2A) ---------------------------------------
# A2A is a route POLICY (`a2a: {}`), not a backend: it marks a route as A2A traffic
# and forwards to the agent's own A2A server (the BYO container serves it on :8080).
# Add this as another route under the listener above:
#
#      - policies:
#          a2a: {}
#          cors:
#            allowOrigins: ["*"]
#            allowHeaders: [content-type, cache-control]
#        backends:
#          - host: localhost:8080
```

Read it as: _"this route carries A2A traffic (`a2a: {}`) and forwards it to the agent's A2A server at `localhost:8080`."_ The ADK agent serves A2A on port `8080` (the same port the BYO container exposes in Kubernetes, per `infra/README.md`), so the `host` backend targets it directly.

This mirrors the upstream agentgateway `a2a` example, where the same `a2a: {}` policy sits above a `host` backend:

```yaml
policies:
  cors:
    allowOrigins:
      - "*"
    allowHeaders:
      - content-type
      - cache-control
  # Mark this route as a2a traffic (a sibling policy of cors, not a field of it)
  a2a: {}
backends:
  - host: localhost:9999
```

## Why does the agent card get rewritten?

An A2A server advertises itself through an **agent card** at `/.well-known/agent.json` (the current A2A/kagent path is `/.well-known/agent-card.json`, as used by the Ops Copilot container in [Chapter 6](../6. Platform/6.3. Platform Agents.md)), which includes the `url` other agents should call. When the gateway proxies A2A, it rewrites that `url` to point back at the gateway. In the upstream example, a card served by an agent on `localhost:9999` comes back through the gateway with its `url` set to the gateway address:

```json
{
  "description": "Just a hello world agent",
  "url": "http://localhost:3000"
}
```

That rewrite is what guarantees future requests keep flowing through the gateway (and its policies) rather than hitting the agent directly and bypassing every guard you configured.

## How do I know A2A traffic is being handled?

With `config.logging.format: json` (or the default log format), the access log tags each request with A2A-specific fields — the method and duration of the A2A call — so you can confirm the policy is active. The upstream example shows a log line like:

```plain
info request gateway=bind/3000 listener=listener0 route=route0 endpoint=localhost:9999
  http.method=POST http.status=200 a2a.method=message/stream duration=2ms
```

The `a2a.method=message/stream` field is the tell that the `a2a: {}` policy classified the request correctly.

!!! note "One listener, many routes"

    The MCP route (5.2) and the A2A route can live under the **same listener** on port `3000` —
    each route matches its own traffic and applies its own policies and backend. That is why the
    A2A block is described as *"another route under the listener above"*.
