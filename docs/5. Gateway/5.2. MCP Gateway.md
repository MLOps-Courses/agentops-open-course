# 5.2. MCP Gateway

## Why put a gateway in front of an MCP server?

In [3.3. MCP](../3. Capabilities/3.3. MCP.md) the agent launched the Ops Copilot MCP server itself, over stdio, as a child process. That is perfect for local development but has limits: the agent must know how to start the server, every agent that wants the tools re-launches its own copy, and there is no place to attach auth, rate limits, or tracing.

Fronting the server with agentgateway fixes all three:

1. **One secured endpoint** — the gateway launches the server once and re-exposes its tools over HTTP, so any number of agents share it.
1. **Federation / multiplexing** — several MCP servers can be aggregated behind one URL and presented to the agent as a single tool namespace.
1. **Policy attach point** — auth, authorization, rate limits, and observability (5.5–5.6) apply at the gateway, not in every agent.

## How does the gateway front the Ops Copilot MCP server?

The active config declares an `mcp` backend with one `target` — the Python `agent.mcp_server` launched over stdio. The gateway runs that command and adapts its stdio transport into HTTP:

```yaml
backends:
  - mcp:
      targets:
        - name: ops-copilot
          stdio:
            cmd: uv
            args: [run, --directory, ../agents/python, python, -m, agent.mcp_server]
```

agentgateway serves the aggregated MCP server two ways:

1. **Streamable HTTP** under `/mcp` — the modern default.
1. **SSE** under `/sse` — for clients that still use Server-Sent Events.

So the Ops Copilot tools that ran over stdio in Chapter 3 are now reachable at `http://localhost:3000/mcp` with CORS and (later) security policies applied.

## How does the agent point at the gateway instead of the raw server?

This is the seam the MCP client was written for. In Chapter 3.3 the toolset launched the server over stdio; to use the gateway you swap the **transport** from stdio to the gateway's HTTP URL — the agent's tools, prompts, and workflow are untouched. The Python client was written with this in mind: `ops_mcp_toolset()` builds an `McpToolset` with `StdioConnectionParams`. Point it at the gateway by switching to an HTTP/streamable connection targeting `http://localhost:3000/mcp` instead of the stdio server params. The docstring of `agents/python/src/agent/mcp_client.py` calls this out: _"point the toolset at the gateway instead of the raw server."_

The tool names the model sees do not change, so no prompt or evaluation has to be rewritten.

## How do I multiplex multiple MCP servers?

Add more entries under `targets`. The gateway aggregates them and exposes every tool through the one endpoint. The active config keeps a commented example ready to uncomment:

```yaml
targets:
  - name: ops-copilot
    stdio:
      cmd: uv
      args: [run, --directory, ../agents/python, python, -m, agent.mcp_server]
  # Multiplex more MCP servers by adding targets, e.g.:
  # - name: time
  #   stdio:
  #     cmd: uvx
  #     args: [mcp-server-time]
```

Targets are not limited to stdio. The gateway can also connect to a remote MCP server over HTTP (`mcp: { host: … }`) or wrap an OpenAPI spec as MCP tools (`openapi:`), so you can federate local and remote tool servers behind a single agent-facing URL. This is the agentgateway `multiplex` example applied to the Ops Copilot.

## How does this map to Kubernetes later?

The same gateway MCP endpoint is what kagent registers in the cluster. `infra/kagent/toolserver.yaml` holds a **`RemoteMCPServer`** custom resource pointing at the gateway's `/mcp` path:

```yaml
apiVersion: kagent.dev/v1alpha2
kind: RemoteMCPServer
metadata:
  name: agentops-tools
  namespace: agentops
spec:
  description: Ops Copilot read-only tools (incidents, service status, runbooks) over MCP.
  url: http://agentgateway.agentops.svc.cluster.local:3000/mcp
  protocol: STREAMABLE_HTTP
  timeout: 30s
```

Locally you point the toolset at `http://localhost:3000/mcp`; in the cluster kagent points a `RemoteMCPServer` at the gateway Service's `/mcp` — same endpoint, same tools, different address. This is covered in [6.4. Platform Tools](../6. Platform/6.4. Platform Tools.md).

!!! tip "Prefer the HTTP path over stdio in the cluster"

    Running the MCP server over stdio inside a platform requires the runtime to ship `uv` and the
    agent source. Fronting it with the gateway and registering a `RemoteMCPServer` avoids that
    coupling — the agent just talks HTTP to a stable URL.
