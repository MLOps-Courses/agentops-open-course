# 5.1. Gateway Setup

## How do I install agentgateway?

You have two options, both pinned to **v1.3.1** for this course. Pick one:

1. **Single binary** — download the release for your platform from the [agentgateway releases](https://github.com/agentgateway/agentgateway/releases), or build from source. It is one self-contained executable, no runtime dependencies.
1. **Container** — pull the canonical image `cr.agentgateway.dev/agentgateway` (the project's own registry; it also mirrors to `ghcr.io/agentgateway/agentgateway`).

!!! warning "Pin the version"

    agentgateway is young and moving fast. Pin **v1.3.1** (binary release tag or image tag) and
    re-verify at build time — the config schema still evolves between minor versions.

## How do I run it locally?

The gateway is driven entirely by a config file passed with `-f` (long form `--file`). Run it from the `infra/` directory so the relative paths in the config resolve correctly:

```bash
cd infra
agentgateway -f agentgateway/config.yaml
```

With the container image, mount the config and publish the ports:

```bash
docker run --rm \
  -p 3000:3000 -p 15000:15000 \
  -v "$PWD/agentgateway:/config" \
  cr.agentgateway.dev/agentgateway -f /config/config.yaml
```

That is it — no Kubernetes, no sidecars. The gateway starts, launches the backends declared in the config, and begins serving on the ports you bound.

## What is the admin UI at :15000?

While the data plane serves traffic on the ports you `bind` (port `3000` in our config), the gateway also exposes an **admin UI on `localhost:15000`**. Open it in a browser to inspect the live configuration, see the listeners and routes, and browse the backends the gateway is fronting. It is the fastest way to confirm the gateway loaded your config as intended.

## What does the config actually look like?

Here is the real, active `infra/agentgateway/config.yaml` from this repo. It fronts the Ops Copilot MCP server (Chapter 3.3) and re-exposes its tools over one secured HTTP endpoint:

```yaml
# yaml-language-server: $schema=https://agentgateway.dev/schema/config
binds:
  - port: 3000
    listeners:
      - routes:
          - policies:
              cors:
                allowOrigins: ["*"]
                allowHeaders:
                  - mcp-protocol-version
                  - content-type
                  - cache-control
                  - mcp-session-id
                exposeHeaders:
                  - Mcp-Session-Id
            backends:
              # The Python MCP server (Ch. 3.3), launched over stdio. agentgateway has
              # no working-dir field, so `uv --directory` runs it from agents/python
              # (paths are relative to where the gateway binary runs — this infra/ dir).
              - mcp:
                  targets:
                    - name: ops-copilot
                      stdio:
                        cmd: uv
                        args: [run, --directory, ../agents/python, python, -m, agent.mcp_server]
```

## How is a config file structured?

Every agentgateway config nests the same four layers, outermost to innermost:

1. **`binds`** — the ports the gateway listens on. Ours binds port `3000`.
1. **`listeners`** — groups of routes under a bind. Our simple case has a single listener.
1. **`routes`** — match traffic and attach `policies` (here, CORS so browser-based MCP clients like the MCP Inspector can connect).
1. **`backends`** — where matched traffic is finally sent. Backend **types** are `mcp`, `ai` (LLM), `host`, `service`, `aws`, `dynamic`, and `internal`.

Read it top-down as: _"listen on 3000, apply these policies to every request, and forward to this MCP backend."_ The Ops Copilot MCP server is launched over **stdio** — the gateway runs the `uv run … python -m agent.mcp_server` command and adapts its stdio transport into HTTP for you.

!!! note "Relative paths run from the gateway's working directory"

    Paths in the config (like `../agents/python`) are relative to **where the binary runs**, not
    to the config file. That is why you launch from `infra/`. agentgateway has no working-directory
    field, so `uv --directory` sets it for the child MCP process.

## How do I know it worked?

Two quick checks:

1. Open `http://localhost:15000` — the admin UI should show one bind, one listener, one route, and the `ops-copilot` MCP backend.
1. Point the [MCP Inspector](https://github.com/modelcontextprotocol/inspector) at the streamable HTTP endpoint `http://localhost:3000/mcp` (or SSE at `/sse`) and list tools — you should see the Ops Copilot tools (`list_incidents`, `get_incident`, …).

The Ch. 5.3–5.5 additions ship as clearly commented blocks in the same file, so the active config stays minimal and valid; you uncomment them as each capability is introduced in the next sections.
