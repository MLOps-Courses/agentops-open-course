# 5.5. Gateway Security

## What can the gateway secure?

Because every MCP, A2A, and LLM call passes through the gateway, it is the natural place to enforce security once, in one config. agentgateway policies cover four layers:

1. **Authentication** — verify who is calling (JWT, API key, basic auth, OIDC/OAuth2).
1. **Authorization** — decide which tools or actions a caller may use (CEL rules on MCP tools).
1. **Rate limiting** — cap request rate locally or globally to protect upstreams and budgets.
1. **Prompt guarding** — reject or mask PII and injection in LLM prompts and responses.

You attach these as `policies` on a route, right next to the CORS policy you already have. This section quotes the commented 5.5 blocks in `infra/agentgateway/config.yaml` and points at the upstream examples for the full feature detail.

## How do I add rate limiting and MCP authorization?

The active config keeps a 5.5 block ready to attach to the MCP route. `localRateLimit` caps the request rate; `mcpAuthorization` gates individual tool calls with CEL rules:

```yaml
# --- Chapter 5.5 — Secure Gateway (rate limit + prompt guard) -----------------
# Attach security policies to a route. `localRateLimit` caps request rate on any
# route; `mcpAuthorization` gates individual MCP tool calls (CEL rules); the
# `ai.promptGuard` block rejects PII in prompts/responses on `ai` backends:
#
#        policies:
#          localRateLimit:
#            - maxTokens: 60
#              tokensPerFill: 60
#              fillInterval: 60s
#          mcpAuthorization:
#            rules:
#              # only these MCP tools reach this caller; the gateway filters the rest
#              # of the read-only tool list out of tools/list and denies them if called
#              - 'mcp.tool.name == "list_incidents"'
#              - 'mcp.tool.name == "get_service_status"'
```

The `localRateLimit` is a token bucket: `maxTokens` is the burst size, `tokensPerFill` refills that many tokens every `fillInterval`. The example allows 60 requests per minute.

The `mcpAuthorization` rules narrow the Ops Copilot's **tool surface at the gateway**. Be precise about what is on that surface: the MCP server exposes only the agent's **read-only** tools (`list_incidents`, `get_service_status`, `get_incident`, `get_runbook`, `search_runbooks`). The **state-changing** actions (`restart_service`, `resolve_incident`) are deliberately kept **in-process** and human-in-the-loop guarded ([4.5. Guardrails](../4. Quality/4.5. Guardrails.md)) — they are never exposed over MCP, so the gateway can neither list nor deny them.

So `mcpAuthorization` decides **which read-only tools a given caller may reach**: the gateway filters any tool not permitted by a rule out of the `tools/list` result and denies it if called. The example above admits only `list_incidents` and `get_service_status` for this caller; add rules to widen the set. This is genuine defence in depth, but split by where each control belongs — the write boundary lives inside the agent (HITL), and the gateway governs read access and gates who may call what.

## How do the authorization rules work?

`mcpAuthorization.rules` are [CEL](https://cel.dev/) expressions — a small language for authorization policy. Each rule permits a tool call when it evaluates true. Rules can combine tool identity with JWT claims, as in the upstream `authorization` example:

```yaml
mcpAuthorization:
  rules:
    # Allow anyone to call 'echo'
    - 'mcp.tool.name == "echo"'
    # Only the test-user can call 'get-sum'
    - 'jwt.sub == "test-user" && mcp.tool.name == "get-sum"'
    # Any authenticated user with the claim `nested.key == value` can access 'get-env'
    - 'mcp.tool.name == "get-env" && jwt.nested.key == "value"'
```

When a rule references `jwt.*`, the caller must present a valid token — which brings in authentication.

## How do I require authentication?

Add a `jwtAuth` policy to the route. It declares the trusted issuer, the accepted audiences, and where to find the signing keys (JWKS). From the upstream `authorization` example:

```yaml
policies:
  jwtAuth:
    issuer: agentgateway.dev
    audiences: [test.agentgateway.dev]
    jwks:
      # Relative to the folder the binary runs from, not the config file
      file: ./manifests/jwt/pub-key
```

With this in place, callers must present a valid JWT, and `mcpAuthorization` rules can reference its claims (`jwt.sub`, `jwt.nested.key`, …). agentgateway also supports **API key** and **basic** auth for simpler setups.

## How do I add browser login with OIDC?

For human-facing routes, the built-in `oidc` policy runs the full OpenID Connect code flow — the gateway redirects to your identity provider, completes the exchange itself, and forwards the authenticated request upstream. From the upstream `oidc` example (Keycloak as the issuer):

```yaml
policies:
  oidc:
    issuer: http://localhost:7080/realms/agentgateway
    clientId: agentgateway-browser
    clientSecret: agentgateway-secret
    redirectURI: http://localhost:3000/oauth/callback
    scopes:
      - profile
      - email
```

After login the standard JWT claims (`jwt.sub`, `jwt.email`) are available to access logs and authorization rules — the same claim surface `jwtAuth` exposes.

## How do I block PII in model traffic?

On an `ai` backend (5.4), attach an `ai.promptGuard` policy. It scans request and response content and rejects (or masks) matches. The config's 5.5 block shows a PII reject on requests:

```yaml
#          ai:
#            promptGuard:
#              request:
#                - regex:
#                    action: reject
#                    rules:
#                      - builtin: email      # built-in PII detector; or `pattern: SSN` for a regex
```

`builtin` detectors ship for common PII (like `email`); `pattern` takes your own regex (like `SSN`). The upstream `ai-prompt-guard` example guards **requests** with two reject rules and returns a clean, OpenAI-compatible JSON error on rejection (a `response:` block guards model output the same way):

```yaml
policies:
  ai:
    promptGuard:
      request:
        - regex:
            action: reject
            rules:
              - pattern: SSN
              - pattern: Social Security
        - regex:
            action: reject
            rules:
              - builtin: email
```

Beyond regex/builtin, the same example demonstrates `azureContentSafety`, `bedrockGuardrails`, `googleModelArmor`, and a moderation `webhook` — swap the rule type to use a managed guard service.

## Can I rate-limit globally across replicas?

Yes. `localRateLimit` is per-gateway-instance; for a shared limit across replicas, use `remoteRateLimit` backed by an external rate-limit service (Envoy's ratelimit server + Redis in the upstream `ratelimiting/global` example). It also exposes a `failureMode` (`failOpen` by default, or `failClosed`) that decides whether requests pass or are denied when the limit service is unreachable. Start with `localRateLimit` locally; reach for `remoteRateLimit` once the gateway runs as a scaled cluster workload.

!!! tip "Layer the policies"

    The strongest posture stacks them on one route: `jwtAuth` (who), `mcpAuthorization` (what),
    `localRateLimit` (how much), and `ai.promptGuard` (what content). The upstream `ratelimiting`
    example combines auth, authorization, and rate limiting on a single MCP route — a good template
    for the Ops Copilot.
