# 0.7. Glossary

## What does each course term mean?

Every term links to the section that introduces it. New to agents? Start with **[Agent](#agent)**, **[Tool](#tool)**, **[MCP](#mcp)**, and **[A2A](#a2a)**, then follow the links into the chapters.

- <a id="a2a"></a>**A2A**: Protocol for discovering agents and exchanging tasks across process boundaries. [3.6. A2A](../3.%20Capabilities/3.6.%20A2A.md#what-is-a2a-and-why-is-it-not-just-mcp-again)
- <a id="a2a-card"></a>**A2A card**: JSON descriptor an A2A server publishes declaring its skills, endpoints, and streaming support. [3.6. A2A](../3.%20Capabilities/3.6.%20A2A.md#what-does-the-agent-card-declare)
- <a id="adk"></a>**ADK**: Google Agent Development Kit: the framework providing the agent runner, sessions, tools, and telemetry. [2.0. Concepts](../2.%20Agents/2.0.%20Concepts.md#what-does-adk-provide)
- <a id="agent"></a>**Agent**: A model-driven loop that selects tools or delegates work toward a goal. [0.1. Agents](../0.%20Overview/0.1.%20Agents.md#what-is-an-ai-agent)
- <a id="agent-skill"></a>**Agent Skill**: A small, discoverable instruction package rooted at SKILL.md, loaded only for a relevant task. [3.2. Skills](../3.%20Capabilities/3.2.%20Skills.md#what-is-an-agent-skill)
- <a id="agentgateway"></a>**agentgateway**: The proxy placed in front of agents, models, and tools that enforces policy, auth, and observability. [5.0. Gateway](../5.%20Gateway/5.0.%20Gateway.md#why-add-a-gateway-after-the-agent-works)
- <a id="agentic-loop"></a>**agentic loop**: The iterative reason-act-observe cycle an agent runs to reach a goal. [0.1. Agents](../0.%20Overview/0.1.%20Agents.md#what-is-the-agentic-loop)
- <a id="agentops"></a>**AgentOps**: Practices for operating agents in production: capabilities, guardrails, HITL, and lifecycle. [0.2. AgentOps](../0.%20Overview/0.2.%20AgentOps.md#what-is-agentops)
- <a id="backoff-and-retry"></a>**backoff and retry**: Retrying transient read failures (cold start, restart, locked SQLite) instead of failing the turn. [4.5. Guardrails](../4.%20Quality/4.5.%20Guardrails.md#how-does-the-agent-survive-transient-failures)
- <a id="byo-agent"></a>**BYO Agent**: A kagent "bring your own" Agent resource where the course ships its own A2A container image. [6.3. Platform Agents](../6.%20Platform/6.3.%20Platform%20Agents.md#what-does-the-byo-agent-declare)
- <a id="checkpoint"></a>**checkpoint**: The per-page verification exercise a learner completes to confirm the result before continuing. [0.0. Course](../0.%20Overview/0.0.%20Course.md#how-should-you-work-through-each-page)
- <a id="cosign"></a>**cosign**: Tool used to verify an image is the exact one CI built and signed. [6.1. Containers](../6.%20Platform/6.1.%20Containers.md#how-do-i-prove-an-image-is-the-one-ci-built)
- <a id="cosine-distance"></a>**cosine distance**: Embedding-similarity measure used by the optional semantic retrieval branch. [3.4. Memory](../3.%20Capabilities/3.4.%20Memory.md#when-do-embeddings-beat-keyword-retrieval)
- <a id="cost-attribution"></a>**cost attribution**: Assigning model and token cost to a session or tool for budgeting. [7.3. Costs](../7.%20Observability/7.3.%20Costs.md#how-do-i-attribute-cost-to-a-session-or-tool)
- <a id="deadline"></a>**deadline**: A bounded time limit wrapping read tools so a slow dependency costs a retry, not a hang. [4.5. Guardrails](../4.%20Quality/4.5.%20Guardrails.md#how-does-the-agent-survive-transient-failures)
- <a id="delegation"></a>**delegation**: In-process transfer of control from a coordinator to a named sub-agent in the same process and session, cheaper and lower-latency than networked A2A but sharing one deployment, failure, and trust boundary. [3.6. A2A](../3.%20Capabilities/3.6.%20A2A.md#what-is-in-process-delegation)
- <a id="evaluation"></a>**Evaluation**: Repeatable measurement of behavior, trajectory, safety, or output quality. [4.4. Evaluations](../4.%20Quality/4.4.%20Evaluations.md#what-is-different-about-an-agent-evaluation)
- <a id="grafana"></a>**Grafana**: Dashboarding UI that visualizes the shipped AgentOps metric and log panels. [7.2. Monitoring](../7.%20Observability/7.2.%20Monitoring.md#which-dashboard-is-shipped)
- <a id="guardrail"></a>**Guardrail**: Deterministic or model-assisted policy at an input, model, tool, or output boundary. [4.5. Guardrails](../4.%20Quality/4.5.%20Guardrails.md#what-is-a-guardrail)
- <a id="hitl"></a>**HITL**: Human-in-the-loop: a human decision inserted before a sensitive action continues. [4.5. Guardrails](../4.%20Quality/4.5.%20Guardrails.md#how-does-human-confirmation-work)
- <a id="kagent"></a>**kagent**: CNCF Sandbox project that manages agents as Kubernetes custom resources. [6.0. Platform](../6.%20Platform/6.0.%20Platform.md#what-does-kagent-own)
- <a id="loki"></a>**Loki**: Log store that receives OTLP logs and backs the dashboard's log panel. [7.2. Monitoring](../7.%20Observability/7.2.%20Monitoring.md#where-do-my-agent-logs-go)
- <a id="long-term-memory"></a>**long-term memory**: Explicit cross-session notes a stable user can save and recall, isolated per user and PII-redacted before the write, distinct from per-session conversation state. [3.4. Memory](../3.%20Capabilities/3.4.%20Memory.md#what-should-an-agent-remember-across-sessions)
- <a id="mcp"></a>**MCP**: Model Context Protocol for discovering and invoking tool or context servers. [3.3. MCP](../3.%20Capabilities/3.3.%20MCP.md#what-problem-does-mcp-solve)
- <a id="mlflow"></a>**MLflow**: Self-hosted store for prompt versions, traces, and evaluations. [7.0. Reproducibility](../7.%20Observability/7.0.%20Reproducibility.md#how-does-mlflow-link-prompt-and-model-evidence)
- <a id="multi-agent"></a>**Multi-agent**: A system where one agent delegates to several specialist agents under a coordinator. [3.7. Multi-Agent](../3.%20Capabilities/3.7.%20Multi-Agent.md#why-would-one-agent-become-several)
- <a id="networkpolicy"></a>**NetworkPolicy**: Kubernetes resource that restricts pod reachability and egress to reduce attack surface. [6.5. Platform Gateway](../6.%20Platform/6.5.%20Platform%20Gateway.md#how-does-network-policy-reduce-reachability)
- <a id="opentelemetry-otel"></a>**OpenTelemetry (OTel)**: Vendor-neutral instrumentation and transport layer for traces, metrics, and logs. [7.1. Tracing](../7.%20Observability/7.1.%20Tracing.md#how-is-adk-telemetry-enabled)
- <a id="otlp"></a>**OTLP**: The OpenTelemetry wire protocol (HTTP/gRPC) the app and gateway use to export telemetry. [7.1. Tracing](../7.%20Observability/7.1.%20Tracing.md#how-is-adk-telemetry-enabled)
- <a id="pii"></a>**PII**: Personally identifiable information, redacted before requests reach the model. [4.5. Guardrails](../4.%20Quality/4.5.%20Guardrails.md#where-is-pii-redacted)
- <a id="prometheus"></a>**Prometheus**: Time-series database that scrapes collector and gateway metrics. [7.2. Monitoring](../7.%20Observability/7.2.%20Monitoring.md#how-are-agent-metrics-produced)
- <a id="prompt-injection"></a>**prompt injection**: Attacker-controlled text that tries to override agent instructions. [5.5. Gateway Security](../5.%20Gateway/5.5.%20Gateway%20Security.md#how-does-the-prompt-guard-work)
- <a id="prompt-registry"></a>**prompt registry**: MLflow-backed registry used to version and compare prompts in the host development/evaluation environment; production images use the committed instruction. [7.0. Reproducibility](../7.%20Observability/7.0.%20Reproducibility.md#how-does-mlflow-link-prompt-and-model-evidence)
- <a id="rag"></a>**RAG**: Retrieval that adds relevant external knowledge to the model context. [3.4. Memory](../3.%20Capabilities/3.4.%20Memory.md#how-does-free-text-retrieval-work)
- <a id="red-metrics"></a>**RED metrics**: Rate, Errors, Duration: bounded request metrics derived from spans under namespace agentops. [7.2. Monitoring](../7.%20Observability/7.2.%20Monitoring.md#how-are-agent-metrics-produced)
- <a id="sbom"></a>**SBOM**: Software Bill of Materials listing an image's components for supply-chain auditing. [6.1. Containers](../6.%20Platform/6.1.%20Containers.md#what-is-an-sbom-for)
- <a id="semantic-retrieval"></a>**semantic retrieval**: Optional embedding-based search enabled only when it beats the keyword baseline. [3.4. Memory](../3.%20Capabilities/3.4.%20Memory.md#when-do-embeddings-beat-keyword-retrieval)
- <a id="session"></a>**session**: Per-conversation runtime state (events, history) owned by the ADK runner. [2.4. Sessions](../2.%20Agents/2.4.%20Sessions.md#what-is-a-session)
- <a id="slo"></a>**SLO**: Service Level Objective tied to an observable user outcome (e.g. 99% error-free requests). [4.3. Metrics](../4.%20Quality/4.3.%20Metrics.md#how-do-you-define-a-useful-slo)
- <a id="sops"></a>**SOPS**: Tool for keeping encrypted secrets in git and decrypting them at deploy time. [6.5. Platform Gateway](../6.%20Platform/6.5.%20Platform%20Gateway.md#how-do-you-keep-secrets-in-git-safely)
- <a id="span"></a>**span**: A single timed unit of work in a trace, emitted by ADK and gateway instrumentation. [7.1. Tracing](../7.%20Observability/7.1.%20Tracing.md#what-does-the-collector-do)
- <a id="spanmetrics"></a>**spanmetrics**: OTel connector that turns spans into request-count and duration metrics. [7.1. Tracing](../7.%20Observability/7.1.%20Tracing.md#what-does-the-collector-do)
- <a id="spotlighting"></a>**spotlighting**: Wrapping attacker-influenceable tool output in a marked prefix so the model treats it as data, not instructions. [4.5. Guardrails](../4.%20Quality/4.5.%20Guardrails.md#how-does-injected-content-reach-the-model-and-what-stops-it)
- <a id="stdio"></a>**stdio**: Default MCP transport where the agent launches the server as a child process. [3.3. MCP](../3.%20Capabilities/3.3.%20MCP.md#which-transports-are-supported)
- <a id="streamable-http"></a>**streamable HTTP**: MCP HTTP transport option for a networked server. [3.3. MCP](../3.%20Capabilities/3.3.%20MCP.md#which-transports-are-supported)
- <a id="token-budget"></a>**token budget**: A per-conversation cap that stops the next model call once a session's accumulated tokens are spent, bounding reasoning work rather than dollars. [7.3. Costs](../7.%20Observability/7.3.%20Costs.md#how-do-i-enforce-a-budget)
- <a id="tool"></a>**Tool**: A typed function or remote capability an agent can invoke. [3.1. Tools](../3.%20Capabilities/3.1.%20Tools.md#what-makes-a-function-a-good-agent-tool)
- <a id="trace"></a>**Trace**: A correlated record of model, tool, gateway, and application work in one request. [7.1. Tracing](../7.%20Observability/7.1.%20Tracing.md#why-trace-an-agent-instead-of-logging-the-final-answer)
- <a id="triage-report"></a>**triage report**: A schema-validated final answer produced by a second entry point for machine consumers such as tickets or dashboards, unlike the conversational agent's prose. [2.3. Instructions](../2.%20Agents/2.3.%20Instructions.md#when-should-the-answer-be-a-schema-instead-of-prose)
- <a id="wif"></a>**WIF**: Workload Identity Federation: maps workload identity to cloud IAM without a static key. [6.5. Platform Gateway](../6.%20Platform/6.5.%20Platform%20Gateway.md#how-does-gke-authenticate-to-vertex)
- <a id="workflow"></a>**Workflow**: A declared graph of steps used instead of one autonomous loop for deterministic control. [3.5. Workflows](../3.%20Capabilities/3.5.%20Workflows.md#what-is-a-workflow-and-why-does-an-agent-need-one)
