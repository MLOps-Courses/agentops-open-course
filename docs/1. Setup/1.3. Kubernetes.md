# 1.3. Kubernetes

## Why does an agents course need Kubernetes?

Building an agent is half the story; _operating_ it is the other half. [Chapter 6](../6. Platform/) runs the Ops Copilot as a real Kubernetes workload with [kagent](https://kagent.dev/) (a CNCF project), and [Chapter 7](../7. Observability/) traces and monitors it there. You do not need a cloud account: everything runs on a **local** cluster on your own machine.

You will not deploy anything in this chapter — the goal is a working local cluster and the two CLIs that drive it.

## What is k3d and why this course uses it?

[k3d](https://k3d.io/) runs [k3s](https://k3s.io/) (a lightweight, certified Kubernetes distribution) inside Docker containers. It starts in seconds, uses little memory, and tears down cleanly — ideal for a laptop. It is the author standard for local clusters and is API-compatible with [kind](https://kind.sigs.k8s.io/), so every `kubectl`, `helm`, and manifest in this course works on either.

## What tools do I install?

Three CLIs, plus the container runtime from [1.2. Containers](./1.2. Containers.md):

- **[k3d](https://k3d.io/)** — creates and manages the local cluster.
- **[kubectl](https://kubernetes.io/docs/reference/kubectl/)** — the Kubernetes command-line client.
- **[helm](https://helm.sh/)** — the package manager used to install kagent and its CRDs in Chapter 6.

Install them with your system package manager (Homebrew, `apt`, …); each project's site lists the exact command.

## How do I create a local cluster?

Create a cluster and confirm the node is ready:

```bash
k3d cluster create local   # create a local Kubernetes cluster named "local"
kubectl get nodes          # verify the cluster is up
```

You should see one node in the `Ready` state:

```text
NAME                  STATUS   ROLES                  AGE   VERSION
k3d-local-server-0    Ready    control-plane,master   30s   v1.31.x+k3s1
```

The course deploys all local projects into this one shared `local` cluster, each in its own namespace, and reaches services over `*.localhost` ingress — for example the deployed agent will answer at `http://agentops.localhost:8080`.

## What runs on the cluster later?

In [Chapter 6](../6. Platform/) you install **kagent** (`v0.10.0-beta3`) via `helm` (or its `kagent install` CLI). kagent manages agents, tools, and models as Kubernetes custom resources and provisions the agent's Deployment and Service from an `Agent` resource pointing at your built image.

!!! info "kagent — CNCF Sandbox project"

    kagent brings agents to Kubernetes as first-class resources. As a Sandbox-stage project its APIs are still `v1alpha*` and move fast, so the course pins chart versions and treats that churn as a versioning lesson in itself.
