# 4.2. Testing

## What do you test in an agent, and what do you not?

You test the **deterministic parts** — the tools, the data layer, the guarded actions, the validation guardrail — the code that runs the same way every time regardless of the model. These are ordinary functions with ordinary inputs and outputs, so they get ordinary unit tests: fast, offline, no provider key. The **non-deterministic part** — does the model call the right tool with the right arguments — is checked separately by evals ([4.4. Evaluations](./4.4. Evaluations.md)), which call the live model and are therefore a slower, manual/CI-optional gate.

This split keeps `mise run test` fast and reproducible: it never touches the network and never spends a token.

```bash
mise run test   # uv run pytest
```

Run **pytest** with branch coverage via `pytest-cov`.

## How do tests avoid mutating the committed database?

This is the central pattern. The Ops Copilot's write actions **change state** — `restart_service` flips a service to operational, `resolve_incident` marks an incident resolved, both append to an audit log. If tests ran against the real `agents/data/incidents.db`, every run would rewrite the committed dataset and the next run would start from a different state. So every test runs against a **fresh, throwaway copy** of the whole dataset, and the agent is pointed at the copy for the duration of the test.

```python
# agents/python/tests/conftest.py
@pytest.fixture(autouse=True)
def isolated_data_dir(tmp_path, monkeypatch):
    """Point the agent at a fresh, disposable copy of the dataset for the duration of a test."""
    destination = tmp_path / "data"
    shutil.copytree(config._DEFAULT_DATA_DIR, destination)  # noqa: SLF001 — test setup mirrors the default
    monkeypatch.setattr(config.settings, "data_dir", destination)
    return destination
```

The fixture is `autouse=True`, so **every** test is isolated automatically. It `copytree`s the bundled data into pytest's `tmp_path` and `monkeypatch`es `config.settings.data_dir` to point there — reverted after the test.

This gives each test a pristine dataset, so a test that resolves `INC-002` cannot affect a test that reads it.

## What does a tool test look like?

It calls the function directly and asserts on the returned data — no model, no mocks. Read tools check the shape and filtering; write actions check the state change **and** the audit entry.

```python
# agents/python/tests/test_actions.py
def test_restart_service_flips_status_and_audits() -> None:
    before = data.get_service("inventory")
    assert before is not None
    assert before["status"] == "down"
    result = actions.restart_service("inventory")
    assert "result" in result
    after = data.get_service("inventory")
    assert after is not None
    assert after["status"] == "operational"
    assert result["audit"]["action"] == "restart_service"
```

Because the store is a throwaway copy, `inventory` reliably starts `down` — the tests assert real preconditions from the seeded fixture instead of guessing, and mutate freely.

## What else is worth a unit test?

The wiring and the guardrail, since they are cheap and catch real regressions:

- **Tool registration** — that the agent registers exactly the tools you expect. `test_agent_registers_tools` asserts `len(root_agent.tools)` equals the sum of the read tools, knowledge tools, and action tools.
- **The HITL flag** — that every action tool really requires confirmation (`tool._require_confirmation is True`), so a refactor cannot silently drop the human-approval gate.
- **The guardrail** — that `validate_actions` blocks a malformed id and passes a valid one. Because the callback only reads the tool name and args, the tests pass a cast-`None` context and call it directly.

That last one is a good example of testing without mocks: the guardrail is written so its inputs are plain values, so the test needs no ADK runtime at all.

!!! tip "Tests run on push, checks on commit"

    lefthook runs `mise run test` in the **pre-push** hook (fast, offline) and `mise run check`
    in **pre-commit**. Evals are in neither — they call the live model, so you run
    `mise run eval` deliberately or in an optional CI job (see [4.4. Evaluations](./4.4. Evaluations.md)).

Unit tests prove the deterministic pieces work. Next, [4.3. Metrics](./4.3. Metrics.md) defines what "working" means for the agent as a whole.
