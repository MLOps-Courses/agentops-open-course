# 4.4. Evaluations

## What is an agent evaluation?

An **evaluation** (an _eval_) runs the real agent — real model, real tools — against recorded cases and scores its behavior. Where a unit test calls a tool function directly ([4.2. Testing](./4.2. Testing.md)), an eval asks the whole loop a question and checks _what the agent did_: which tools it called, with which arguments, and what it finally answered. It is how you measure the non-deterministic half of quality from [4.3. Metrics](./4.3. Metrics.md).

Evals **call the live model**, so they are slower, need a provider key, and vary run to run. They are therefore a deliberate, manual/CI-optional gate — explicitly **not** part of `mise run test`, which stays fast and offline.

## What does an eval case look like?

Each case fixes a user prompt, the **expected tool trajectory** (the tools and arguments the agent should call), and a reference final response. A single case from the Ops Copilot's core set:

=== "Python"

    ```json
    // agents/python/evals/ops.evalset.json
    {
      "eval_id": "inventory-status",
      "conversation": [
        {
          "invocation_id": "inv-inventory-status",
          "user_content": {
            "role": "user",
            "parts": [{ "text": "What is the status of the inventory service?" }]
          },
          "final_response": {
            "role": "model",
            "parts": [{ "text": "The inventory service is currently down." }]
          },
          "intermediate_data": {
            "tool_uses": [
              { "name": "get_service_status", "args": { "name": "inventory" } }
            ],
            "intermediate_responses": []
          }
        }
      ],
      "session_input": { "app_name": "agentops_agent", "user_id": "eval", "state": {} }
    }
    ```

=== "Go"

    ```text
    # agents/go/evals/README.md
    Evaluation datasets for the Go agent (Chapter 4.4). Add evaluation cases here and run them
    against the agent to measure task success, tool accuracy, latency, and cost.
    ```

The `tool_uses` block is the heart of the case: given the question about `inventory`, the agent must call `get_service_status` with `{"name": "inventory"}` — the exact deterministic contract over the bundled dataset. The other cases add a second hop (`get_incident` then `get_runbook` for a fix recommendation), exercising a real triage trajectory.

!!! info "Eval tooling is Python-first"

    The `adk eval` CLI, the `*.evalset.json` format, and rubric scoring ship with the **Python**
    ADK. The **Go** track keeps its eval datasets under `agents/go/evals/` and runs them
    programmatically against the agent; there is no `adk eval` binary on the Go track today. The
    *concepts* on this page — fixed cases, tool-trajectory gating — apply to both; the concrete
    tool below is Python's.

## How do I run the evals?

Through the `eval` task on the Python track, which points `adk eval` at the eval set and a config file:

=== "Python"

    ```bash
    mise run eval
    # uv run adk eval src/agent evals/ops.evalset.json --config_file_path evals/test_config.json
    ```

=== "Go"

    ```bash
    # No adk eval CLI on the Go track — drive the eval dataset through the agent programmatically.
    ```

Because it hits the live model, it needs a provider key in `.env` (native Gemini, per Chapter 2). Run it before a release, or in a separate CI job — never inline with the fast unit suite.

## Why gate on the tool trajectory instead of the response text?

Because the trajectory is **deterministic and meaningful**, and the response text is not. The pass criteria live in a tiny config file:

=== "Python"

    ```json
    // agents/python/evals/test_config.json
    {
      "criteria": {
        "tool_trajectory_avg_score": 1.0
      }
    }
    ```

=== "Go"

    ```text
    # The same idea applies: score the tool trajectory, not the exact wording of the answer.
    ```

`tool_trajectory_avg_score: 1.0` demands the agent call **exactly** the right tools with the right arguments — a contract the fixed dataset makes reproducible. ADK can also compute `response_match_score` (ROUGE overlap between the agent's answer and the reference text), but the course **does not gate on it**: generative wording varies run to run, so a perfectly correct answer can score low just for phrasing it differently. Trajectory is the tightest deterministic proxy for correctness here, so it is the gate; response match stays a soft signal you read, not a pass/fail bar.

!!! tip "Grow the set as the agent grows"

    Add more `*.evalset.json` files under `evals/` as you add capabilities — a new tool or a new
    workflow deserves a new case. Keep each case's trajectory exact so the 1.0 gate stays honest,
    and the eval set doubles as executable documentation of what the agent is supposed to do.

Evals catch _wrong_ behavior. Next, [4.5. Guardrails](./4.5. Guardrails.md) prevents _unsafe_ behavior before it happens.
