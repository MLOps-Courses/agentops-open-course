# 4.1. Linting

## Why lint an agent's code?

Linters catch the bugs a type checker misses and enforce one consistent style, so reviews argue about behavior, not formatting. For an agent this matters twice over: the tool functions are **prompt material** (their names, docstrings, and structure are read by the model and by ADK's schema generation), and the write actions touch state, so dead code, a swallowed error, or an unsafe SQL string is a real risk. Both tracks run a linter in `mise run check` and a formatter in `mise run format`, and lefthook runs them on every commit.

## How do I lint and format the Python agent?

With **ruff** — one fast tool for both linting and formatting. The linter runs a broad, curated rule set selected in `pyproject.toml`; a few of the enabled families:

=== "Python"

    ```toml
    # agents/python/pyproject.toml
    [tool.ruff.lint]
    select = [
      "B", # flake8-bugbear
      "E", # pycodestyle errors
      "F", # pyflakes
      "I", # isort
      "PTH", # flake8-use-pathlib
      "RUF", # ruff-specific rules
      "S", # flake8-bandit
      "SIM", # flake8-simplify
      "UP", # pyupgrade
      # ...and more: ARG, ASYNC, C4, N, PERF, PT, RET, TID
    ]
    ```

=== "Go"

    ```yaml
    # agents/go/.golangci.yml — see the Go tab below for the enabled linters
    ```

Note `S` (flake8-bandit) is a **security** rule set — it flags things like unparameterized SQL — and `PT` enforces idiomatic pytest style. Tests relax a couple of rules where they are noise rather than signal:

=== "Python"

    ```toml
    # agents/python/pyproject.toml
    [tool.ruff.lint.per-file-ignores]
    "tests/**" = [
      "S101", # assert
      "T201", # print statement allowed in tests
    ]
    ```

=== "Go"

    ```yaml
    # agents/go/.golangci.yml — gosec is likewise relaxed in _test.go files
    linters:
      # ... default + enabled linters ...
      exclusions:
        rules:
          - path: _test\.go
            linters:
              - gosec
    ```

Run them:

=== "Python"

    ```bash
    mise run format       # ruff check --select=I --fix  &&  ruff format
    mise run check:lint   # uv run ruff check --force-exclude .
    ```

=== "Go"

    ```bash
    mise run format       # goimports -w  &&  gofumpt -w
    mise run check:lint   # go tool golangci-lint run ./...
    ```

## How do I lint and format the Go agent?

With **golangci-lint**, the standard aggregate runner. It layers extra linters on top of the default set in `.golangci.yml` — a representative slice:

=== "Python"

    ```toml
    # agents/python/pyproject.toml — see the Python tabs above for ruff
    ```

=== "Go"

    ```yaml
    # agents/go/.golangci.yml
    linters:
      default: standard
      enable:
        - bodyclose      # every HTTP/response body is closed
        - errcheck       # no ignored error returns
        - errorlint      # correct error wrapping/comparison
        - gosec          # security issues (e.g. SQL, subprocess)
        - revive         # style and best-practice checks
        - sqlclosecheck  # sql.Rows/Stmt are closed
        - testifylint    # idiomatic testify assertions
        - unparam        # unused function parameters
        - unused         # dead code
    ```

`errcheck` and `sqlclosecheck` directly protect the data layer — the incidents tools open `sql.Rows` that **must** be closed — and `gosec` guards the write actions. Formatting is handled by `gofumpt` (stricter `gofmt`) plus `goimports`, both declared under `formatters` in the same file.

## Why is govet's fieldalignment disabled?

Because it optimizes for machine layout at the expense of readable code. `fieldalignment` reorders struct fields to minimize padding, which fights the natural, documented field order in types like `restartServiceInput`. The config keeps **every other** `go vet` analyzer via `enable-all: true` and disables only this one, with the reason recorded inline:

=== "Python"

    ```toml
    # (no equivalent on the Python track — ty and ruff cover this ground)
    ```

=== "Go"

    ```yaml
    # agents/go/.golangci.yml
    govet:
      enable-all: true
      # fieldalignment is not part of Go's default vet — it optimizes struct padding at the
      # cost of readable, conventional field order. We keep every other govet analyzer.
      disable:
        - fieldalignment
    ```

That is the pattern for every suppression in this course: **specific and documented**, never a blanket `# noqa` or `//nolint` — a disabled check should always say why.

## What formats the config and docs?

**dprint** — the single formatter for the non-code files: JSON, TOML, YAML, and Markdown (including these course pages). It is configured at the repo root in `dprint.json` with a shared line width and one wasm plugin per language:

=== "Python"

    ```jsonc
    // dprint.json (repo root) — formats *.json, *.toml, *.yaml, *.md
    {
      "lineWidth": 120,
      "markdown": { "textWrap": "never" },
      "plugins": [
        "https://plugins.dprint.dev/dockerfile-0.4.1.wasm",
        "https://plugins.dprint.dev/g-plane/pretty_yaml-v0.6.0.wasm",
        "https://plugins.dprint.dev/json-0.22.0.wasm",
        "https://plugins.dprint.dev/markdown-0.22.1.wasm",
        "https://plugins.dprint.dev/toml-0.7.0.wasm"
      ]
    }
    ```

=== "Go"

    ```jsonc
    // dprint.json is repo-wide and language-agnostic — the same config formats both tracks' YAML/TOML/JSON.
    ```

At the root, `mise run format` fans out to dprint plus both tracks' formatters, and `mise run check` runs `dprint check` alongside the per-track linters. lefthook runs the matching formatter on staged files before each commit, so the tree stays clean without anyone remembering to run it.

!!! tip "Format before you check"

    The pre-commit hook is ordered deliberately: formatters run and **restage** first, then
    `check` reads the fixed files. Locally, run `mise run format` before `mise run check` for the
    same effect — a formatting-only diff never fails the lint gate.

With style and types enforced, the next gate is behavior: [4.2. Testing](./4.2. Testing.md).
