# 4.3. Metrics

## Why do agents need their own quality metrics?

Because "the code passes its tests" says nothing about whether the **agent** is any good. The tools can be individually correct while the agent picks the wrong one, hallucinates an incident, takes ten seconds, or burns a fortune in tokens to answer a one-line question. Agent quality is a property of the whole loop — model, prompt, tools, and data together — so you measure it with metrics that describe **behavior**, not just unit correctness.

This page is the conceptual map. The two chapters that follow make it concrete: [4.4. Evaluations](./4.4. Evaluations.md) measures behavior offline against a fixed dataset, and [7.1. Tracing](../7. Observability/7.1. Tracing.md) measures it in production from OpenTelemetry spans.

## What are the metrics that matter?

Six recur across every agent, roughly in order of "did it work" to "at what cost":

- **Task success** — did the agent actually accomplish what the user asked? The end-to-end outcome, not intermediate steps.
- **Tool-call accuracy / trajectory** — did it call the right tools, in a sensible order, with the right arguments? For a tool-using agent this is the single most predictive signal of quality, because a correct trajectory almost always yields a correct answer, and it is deterministic enough to gate on.
- **Groundedness** — is the answer supported by what the tools returned, or did the model invent it? For the Ops Copilot the operating rule is explicit: _never invent incidents, services, or statuses_, and _cite the runbook you used_.
- **Latency** — how long from question to answer, including every tool round-trip. On-call users feel this directly.
- **Token cost** — input + output tokens per task, which maps to money and to context-window pressure.
- **Safety** — did the agent respect its guardrails: no unapproved state changes, malformed inputs rejected, no prompt-injection payload obeyed.

## How does the Ops Copilot measure each one?

Two sources cover all six: **evals** (offline, deterministic, over the bundled dataset) and **traces** (online, from OTel spans emitted as the agent runs). The split matters — evals tell you whether a change is _better_ before you ship it; traces tell you what is _actually happening_ once you have.

| Metric               | How the Ops Copilot measures it                                                                                                                                     |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Task success         | Eval case reference response + human/LLM spot-check; in prod, user feedback ([7.4](../7. Observability/7.4. Feedback.md))                                           |
| Tool-call trajectory | **`tool_trajectory_avg_score` gated at 1.0** in `adk eval` — the exact tool+args sequence in `ops.evalset.json` ([4.4](./4.4. Evaluations.md))                      |
| Groundedness         | Grounding rule in the instruction + guardrails; runbook citation checked in eval cases; drift watched online ([7.5](../7. Observability/7.5. Online Evaluation.md)) |
| Latency              | OpenTelemetry span durations per invocation and per tool call ([7.1](../7. Observability/7.1. Tracing.md), [7.2](../7. Observability/7.2. Monitoring.md))           |
| Token cost           | Token counts on model spans, aggregated into budgets/KPIs ([7.3](../7. Observability/7.3. Costs.md))                                                                |
| Safety               | Guardrail blocks + HITL approvals, audit log, unit tests over `validate_actions` ([4.5](./4.5. Guardrails.md))                                                      |

The table is the reason the eval set gates on **tool trajectory** rather than on fuzzy text match: for this agent, the trajectory is the metric with the tightest link to correctness _and_ the one you can check deterministically offline. Everything softer — task success, groundedness — is confirmed by the reference response and, in production, by traces and feedback.

## Which metrics gate a release, and which do you just watch?

Split them by determinism:

- **Gates** (must pass before merge): tool trajectory at 1.0, plus the whole deterministic test suite from [4.2. Testing](./4.2. Testing.md). These are reproducible, so a red result is always a real regression.
- **Watched** (tracked, alerted, budgeted — not a hard gate on every commit): latency, token cost, groundedness drift. These vary run to run and depend on the live model, so you set thresholds and alerts ([7.2. Monitoring](../7. Observability/7.2. Monitoring.md), [7.3. Costs](../7. Observability/7.3. Costs.md)) rather than failing a build on a single slow response.

!!! info "Offline first, online always"

    Measure what you can offline and deterministically — it is cheap, reproducible, and blocks
    regressions before they ship. Then keep measuring online, because the live model, real prompts,
    and real data surface behavior no fixed dataset can. Chapter 7 is the online half of this page.

With the metrics defined, [4.4. Evaluations](./4.4. Evaluations.md) shows how `adk eval` turns the tool-trajectory metric into an automated gate.
