# 4.0. Typing

## Why does type safety matter for an agent?

An agent is a program that hands **untrusted, model-chosen values** to your code. The model decides which tool to call and fills in the arguments; your tool returns data that flows straight back into the model's next decision. Every tool is a boundary between a probabilistic caller and deterministic code, so a wrong type is not a compile-time curiosity — it is a runtime failure the model may then try to "work around" by hallucinating.

Static typing is the cheapest guardrail you have. It lets you:

- **Parse tool inputs into trusted types at the boundary** — a `str` incident id, a typed struct — instead of passing raw `dict`/`any` around.
- **Encode invariants once**, so a tool that promises `{"count": int, "incidents": [...]}` cannot silently return something else.
- **Fail fast at build time**, before an eval or a live run spends tokens discovering the bug.

The Python track treats a clean type check as a **correctness requirement**, wired into `mise run check` next to linting and tests.

## How do I type-check the Python agent?

With **ty**, Astral's fast type checker (pinned pre-1.0 in `pyproject.toml`). It reads the same code the agent runs and verifies annotations across the package. The target environment is declared once in `pyproject.toml`:

```toml
# agents/python/pyproject.toml
[tool.ty.environment]
python-version = "3.13"
```

Run it through the canonical task, which the aggregate `mise run check` depends on:

```bash
mise run check:types   # uv run ty check
```

The Python agent leans on type hints everywhere: `settings.model: str`, tools annotated `def get_incident(incident_id: str) -> dict[str, Any]`, and `from __future__ import annotations` so annotations stay cheap. Those hints are not decoration — ADK also reads them to build the JSON schema the model sees (see [3.1. Tools](../3. Capabilities/3.1. Tools.md)), so a precise type improves both safety and prompt quality.

## Where do types actually protect the agent?

At the **write boundary** — the guarded actions. The model can ask to `resolve_incident` with any string it likes; typing narrows what reaches your business logic, and a validation guardrail narrows it further: the callback parses arguments out of `args: dict[str, Any]` and coerces with `str(args.get("incident_id", ""))` before matching them against a pattern.

The untrusted value is turned into a known type at the edge, then checked, before it ever touches the data store. That layered discipline — parse to a type, then validate the value — is the subject of [4.5. Guardrails](./4.5. Guardrails.md).

!!! tip "Type-check in the loop, not at the end"

    `mise run check` runs types, lint, and audit in parallel, and lefthook runs `check`
    on every commit. Keep the type check green as you write — a red `ty check` blocks the commit,
    which is exactly when the fix is cheapest.
