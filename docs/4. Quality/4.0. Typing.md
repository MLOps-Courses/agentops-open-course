# 4.0. Typing

## Why does type safety matter for an agent?

An agent is a program that hands **untrusted, model-chosen values** to your code. The model decides which tool to call and fills in the arguments; your tool returns data that flows straight back into the model's next decision. Every tool is a boundary between a probabilistic caller and deterministic code, so a wrong type is not a compile-time curiosity — it is a runtime failure the model may then try to "work around" by hallucinating.

Static typing is the cheapest guardrail you have. It lets you:

- **Parse tool inputs into trusted types at the boundary** — a `str` incident id, a typed struct — instead of passing raw `dict`/`any` around.
- **Encode invariants once**, so a tool that promises `{"count": int, "incidents": [...]}` cannot silently return something else.
- **Fail fast at build time**, before an eval or a live run spends tokens discovering the bug.

Both tracks treat a clean type check as a **correctness requirement**, wired into `mise run check` next to linting and tests.

## How do I type-check the Python agent?

With **ty**, Astral's fast type checker (pinned pre-1.0 in `pyproject.toml`). It reads the same code the agent runs and verifies annotations across the package. The target environment is declared once in `pyproject.toml`:

=== "Python"

    ```toml
    # agents/python/pyproject.toml
    [tool.ty.environment]
    python-version = "3.13"
    ```

=== "Go"

    ```go
    // Go has no separate type-checker config: the compiler is the type checker.
    // agents/go/go.mod pins the language version the toolchain type-checks against.
    // go.mod
    // go 1.26.0
    ```

Run it through the canonical task, which the aggregate `mise run check` depends on:

=== "Python"

    ```bash
    mise run check:types   # uv run ty check
    ```

=== "Go"

    ```bash
    mise run check:build   # go build ./...  — compilation is the type check
    ```

The Python agent leans on type hints everywhere: `settings.model: str`, tools annotated `def get_incident(incident_id: str) -> dict[str, Any]`, and `from __future__ import annotations` so annotations stay cheap. Those hints are not decoration — ADK also reads them to build the JSON schema the model sees (see [3.1. Tools](../3. Capabilities/3.1. Tools.md)), so a precise type improves both safety and prompt quality.

## How does Go enforce types?

The **compiler** does it up front — Go will not build code that misuses a type, so `mise run check:build` (`go build ./...`) is a hard type gate on every package. On top of that, `golangci-lint` runs the standard **`go vet`** analyzers with `enable-all`, catching correctness issues the compiler permits (bad `printf` verbs, unreachable code, lost struct tags):

=== "Python"

    ```toml
    # agents/python/pyproject.toml — ty checks the annotated source
    [tool.ty.environment]
    python-version = "3.13"
    ```

=== "Go"

    ```yaml
    # agents/go/.golangci.yml
    linters:
      settings:
        govet:
          enable-all: true
          # fieldalignment is not part of Go's default vet — it optimizes struct padding at the
          # cost of readable, conventional field order. We keep every other govet analyzer.
          disable:
            - fieldalignment
    ```

The Go tools express their contracts as **typed structs with `json` tags** — `restartServiceInput{ Name string }`, `resolveIncidentInput{ IncidentID string }` — and ADK decodes the model's arguments into those structs. A field the model cannot fill correctly is a decode error, not a silent `nil`.

## Where do types actually protect the agent?

At the **write boundary** — the guarded actions. The model can ask to `resolve_incident` with any string it likes; typing narrows what reaches your business logic, and a validation guardrail narrows it further:

- **Python** parses arguments out of the callback's `args: dict[str, Any]` and coerces with `str(args.get("incident_id", ""))` before matching them against a pattern.
- **Go** decodes into `resolveIncidentInput` and type-asserts `args["incident_id"].(string)` in the callback.

In both cases the untrusted value is turned into a known type at the edge, then checked, before it ever touches the data store. That layered discipline — parse to a type, then validate the value — is the subject of [4.5. Guardrails](./4.5. Guardrails.md).

!!! tip "Type-check in the loop, not at the end"

    `mise run check` runs types, lint, and (Python) audit in parallel, and lefthook runs `check`
    on every commit. Keep the type check green as you write — a red `ty check` or a failed
    `go build` blocks the commit, which is exactly when the fix is cheapest.
