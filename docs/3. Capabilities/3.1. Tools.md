# 3.1. Tools

## What is a function tool?

A **tool** is a function the model can call to observe or change the world. Without tools an agent is just a chatbot; with them it can query a database, read a runbook, or restart a service. In ADK, a tool is an ordinary typed function — you write the function, ADK exposes its signature to the model, and when the model decides to call it, ADK runs it and feeds the result back into the loop.

The Ops Copilot starts with three read-only tools over the bundled `incidents.db`:

- `list_incidents` — list incidents, most recent first, optionally filtered.
- `get_incident` — fetch one incident by id, including its runbook slug.
- `get_service_status` — a service's current status plus its open incidents.

The guarded write actions (`restart_service`, `resolve_incident`) that require human approval arrive in [4.5. Guardrails](../4. Quality/4.5. Guardrails.md); this page is the read-only foundation.

## How do I write a tool?

Write a plain, typed function with a clear description, then register it on the agent. ADK reads the function's shape to build the schema the model sees.

```python
# agents/python/src/agent/tools.py
def list_incidents(status: str = "", service: str = "") -> dict[str, Any]:
    """List incidents on the platform, most recent first.

    Args:
        status: Optional filter — one of ``open``, ``investigating``, or ``resolved``.
            Leave empty to list incidents of every status.
        service: Optional service name to filter by (e.g. ``checkout``). Leave empty for all.

    Returns:
        A dict with ``count`` and an ``incidents`` list (id, service, title, severity, status).
    """
    rows = data.list_incidents(status=status or None, service=service or None)
    incidents = [
        {
            "id": row["id"],
            "service": row["service"],
            "title": row["title"],
            "severity": row["severity"],
            "status": row["status"],
        }
        for row in rows
    ]
    return {"count": len(incidents), "incidents": incidents}


def get_incident(incident_id: str) -> dict[str, Any]:
    """Get the full details of one incident by its id.

    Args:
        incident_id: The incident identifier, e.g. ``INC-001``.

    Returns:
        ``{"incident": {...}}`` with the record (including ``runbook`` and ``summary``),
        or an ``error`` if unknown.
    """
    incident = data.get_incident(incident_id)
    if incident is None:
        return {"error": f"No incident found with id {incident_id!r}."}
    return {"incident": incident}


def get_service_status(name: str) -> dict[str, Any]:
    """Get the current status of a service and its open incidents.

    Args:
        name: The service name, e.g. ``checkout`` or ``inventory``.

    Returns:
        ``{"service": {...}, "open_incidents": [...]}``, or an ``error`` if unknown.
    """
    service = data.get_service(name)
    if service is None:
        known = ", ".join(row["name"] for row in data.list_services())
        return {"error": f"No service named {name!r}. Known services: {known}."}
    open_incidents = [
        {"id": row["id"], "title": row["title"], "severity": row["severity"], "status": row["status"]}
        for row in data.list_incidents(service=name)
        if row["status"] != "resolved"
    ]
    return {"service": service, "open_incidents": open_incidents}


# The tools registered on the Ops Copilot. Guarded actions (restart/resolve) join in Ch. 4.5.
ALL_TOOLS: list[ToolUnion] = [list_incidents, get_incident, get_service_status]
```

The tools are plain functions passed straight into `Agent(tools=[...])`; ADK auto-wraps each as a `FunctionTool`.

## How does ADK turn a function into a schema the model sees?

The model never sees your code — it sees a **JSON schema** describing each tool's name, purpose, and parameters. ADK generates that schema from your code, from **type hints and the Google-style docstring**. The parameter names and annotations become the schema's properties; the docstring summary becomes the tool description and the `Args:` lines document each parameter. That is why the tools carry full docstrings: they are prompt material, not just developer notes.

The tool returns a plain dict so the result serializes cleanly back into the conversation. Keep names and descriptions precise — they are the contract the model reasons over.

## Where do the tools get their data?

From a separate **data-access layer** (`data.py`), kept apart from the tool functions so it can be unit-tested directly against `agents/data/incidents.db`. The tools stay thin — they shape data-layer results into model-friendly output; the SQL lives one layer down.

```python
# agents/python/src/agent/data.py
@contextmanager
def _connect() -> Iterator[sqlite3.Connection]:
    """Open a read/write connection with row access by column name; always closed."""
    connection = sqlite3.connect(db_path())
    connection.row_factory = sqlite3.Row
    try:
        yield connection
    finally:
        connection.close()


def list_incidents(status: str | None = None, service: str | None = None) -> list[dict[str, Any]]:
    """Return incidents, newest first, optionally filtered by status and/or service."""
    query = "SELECT * FROM incidents"
    clauses: list[str] = []
    params: list[str] = []
    if status is not None:
        clauses.append("status = ?")
        params.append(status)
    if service is not None:
        clauses.append("service = ?")
        params.append(service)
    if clauses:
        query += " WHERE " + " AND ".join(clauses)
    query += " ORDER BY opened_at DESC"
    with _connect() as connection:
        return [dict(row) for row in connection.execute(query, params)]
```

The data layer binds every user value with a `?` placeholder — no string concatenation of inputs, so there is no SQL-injection surface even though the model chooses the filter values. The state is a module-level connection helper, which makes the dependency obvious and easy to test.

## What about OpenAPI toolsets?

ADK can also generate a whole toolset from an **OpenAPI specification** — point it at a spec and every operation becomes a callable tool, no per-endpoint code. This is how you would later wrap a real services API instead of hand-writing tools. When an API is not described by a spec, you can also expose it through an MCP server ([3.3. MCP](./3.3. MCP.md)) and consume that instead.

For the Ops Copilot the three hand-written function tools are enough, and they keep the course fully offline. Next, teach the agent to fetch procedures on demand with [3.2. Skills](./3.2. Skills.md).
