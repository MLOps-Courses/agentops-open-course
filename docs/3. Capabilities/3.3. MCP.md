# 3.3. MCP

## What is the Model Context Protocol?

The **Model Context Protocol (MCP)** is an open standard for connecting agents to tools that live in their **own process**. Instead of compiling a tool into the agent, you run an MCP **server** that exposes tools over a transport (stdio, HTTP), and any MCP **client** — any agent, in any language — can discover and call them. It is the USB-C of agent tooling: define a capability once, plug it into anything.

For the Ops Copilot this matters because the same MCP server backs the agent and later ([5.2. MCP Gateway](../5. Gateway/5.2. MCP Gateway.md)) sits behind agentgateway with no changes to the agent.

!!! info "AAIF project"

    MCP is a flagship **[AAIF](https://aaif.io/)** (Agentic AI Foundation) standard and an anchor of the open agent ecosystem this course promotes. The course treats it as first-class: exposed here in Chapter 3, then federated through the AAIF's [agentgateway](../5. Gateway/) in Chapter 5.

## What does the MCP server look like?

A handful of lines. The Ops Copilot server re-exposes the read-only tools and the knowledge tools over stdio using **FastMCP**, the server helper from the Python MCP SDK:

```python
# agents/python/src/agent/mcp_server.py
from mcp.server.fastmcp import FastMCP

from . import memory, tools

mcp = FastMCP("ops-copilot")

# Re-expose the read-only tools over MCP. FastMCP derives each tool's schema from the same
# type hints and docstrings the ADK function tools already carry.
for _tool in (
    tools.list_incidents,
    tools.get_incident,
    tools.get_service_status,
    memory.get_runbook,
    memory.search_runbooks,
):
    mcp.add_tool(_tool)


def main() -> None:
    """Run the MCP server over stdio (for an MCP client or the gateway)."""
    mcp.run()


if __name__ == "__main__":
    main()
```

The server reuses the **same functions** as the in-process tools ([3.1. Tools](./3.1. Tools.md)) and knowledge tools ([3.4. Memory](./3.4. Memory.md)). FastMCP derives each tool's JSON schema from the same type hints and docstrings ADK already reads, so there is one source of truth for a tool whether it is called in-process or over MCP. The guarded write actions stay in-process; only the read-only tools are re-exposed.

## How do I run the server?

There is a `mise` task for it — it runs the server over stdio, ready for a client or the gateway to launch:

```bash
cd agents/python
mise run mcp   # uv run python -m agent.mcp_server
```

You rarely run it by hand: the client below launches it for you as a subprocess and talks to it over the process's stdin/stdout.

## How does an agent consume the MCP server?

Through an **MCP toolset**: it launches the server, discovers its tools, and adapts them into ADK tools the agent can call — no change to the agent beyond adding the toolset.

```python
# agents/python/src/agent/mcp_client.py
import sys

from google.adk.tools.mcp_tool.mcp_session_manager import StdioConnectionParams
from google.adk.tools.mcp_tool.mcp_toolset import McpToolset
from mcp import StdioServerParameters


def ops_mcp_toolset() -> McpToolset:
    """Return a toolset that launches and talks to the Ops Copilot MCP server over stdio."""
    return McpToolset(
        connection_params=StdioConnectionParams(
            # Use the current interpreter so it works inside the project's virtualenv.
            server_params=StdioServerParameters(command=sys.executable, args=["-m", "agent.mcp_server"]),
        ),
    )
```

It defines the command that starts the server, then wraps it in a toolset (`McpToolset`). Add the returned toolset to the agent's tools and its MCP tools appear alongside the native ones — the model cannot tell the difference.

## Why put tools behind MCP?

Because MCP is **language-neutral**. The tools are defined once, in the `agent.mcp_server`, and any MCP client consumes them by launching that process. The protocol on the wire is identical regardless of who wrote the client, so:

- You maintain a tool in **one** place, behind a stable contract.
- A tool exposed over MCP is instantly usable from any agent or framework that speaks the protocol.
- The client does not care whether the server is a local subprocess or, later, a remote endpoint behind the gateway.

That last point is the whole design. The toolset is a **seam**: today it points at the raw server over stdio; in [5.2. MCP Gateway](../5. Gateway/5.2. MCP Gateway.md) you repoint the transport at agentgateway and multiplex many MCP servers behind one secured endpoint, with no change to the agent code.
