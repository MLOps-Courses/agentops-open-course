# 3.0. Packaging

## Why package the agent at all?

Because from this chapter on, the Ops Copilot stops being a single script and grows real capabilities — tools, skills, MCP, memory, workflows, delegation. Each lands in its own module so it can be built, tested, and reasoned about in isolation. A clean package layout is what lets the rest of the course add a capability per page without the code turning into a tangle.

Both tracks share one dataset (`agents/data/`) and one shape: a typed `config` at the boundary, a `data` access layer, one module per capability, and a thin entry point that wires them together. Only the language idioms differ.

## How is the Python track laid out?

As a standard **src-layout uv project**. The importable package is `agent`, and it lives under `src/` so tests run against the installed package, not stray files in the working directory:

```text
agents/python/
├── pyproject.toml        # project metadata, deps, tool config
├── mise.toml             # install / format / check / test / run tasks
└── src/
    └── agent/
        ├── __init__.py
        ├── config.py     # typed settings (model, data dir)
        ├── agent.py      # the Ops Copilot agent (Ch. 2)
        ├── data.py       # data-access layer over incidents.db
        ├── tools.py      # function tools (3.1)
        ├── skills.py     # agent skills (3.2)
        ├── mcp_server.py # MCP server (3.3)
        ├── mcp_client.py # MCP client toolset (3.3)
        ├── memory.py     # memory / RAG (3.4)
        ├── workflow.py   # graph workflow (3.5)
        ├── delegation.py # sub-agents / A2A (3.6)
        ├── actions.py    # guarded mock actions (4.5)
        ├── guardrails.py # safety callbacks (4.5)
        └── telemetry.py  # OpenTelemetry setup (7.1)
```

The `pyproject.toml` declares the uv build backend and, crucially, tells it where the package root and name are:

```toml
# agents/python/pyproject.toml
[project]
name = "agentops-agent"
version = "0.1.0"
description = "The AgentOps Open Course reference agent (Python track), built with Google ADK 2.0."
readme = "README.md"
requires-python = ">=3.13"
license = "MIT" # the code license text lives at agents/LICENSE (shared by both tracks)
authors = [{ name = "Médéric HURIER", email = "agentops-open-course@fmind.dev" }]
dependencies = [
  "google-adk>=2.3.0", # native Gemini in Ch. 2–4; other providers via agentgateway in Ch. 5 (no LiteLLM)
  "mcp>=1.16.0", # Model Context Protocol server (FastMCP) + client (McpToolset), Ch. 3.3
  "pydantic-settings>=2.7.1",
]

[build-system]
requires = ["uv_build>=0.9.0,<0.12"]
build-backend = "uv_build"

[tool.uv.build-backend]
module-root = "src"
module-name = "agent"
```

`module-root = "src"` and `module-name = "agent"` are what make `import agent` and `python -m agent.mcp_server` resolve. The runtime dependencies stay deliberately small: ADK itself, the MCP SDK, and `pydantic-settings` for typed configuration.

## How is the Go track laid out?

As a standard **Go module** with the public entry point in `cmd/` and everything else hidden under `internal/`, so nothing leaks into another project's import graph:

```text
agents/go/
├── go.mod                # module path + direct dependencies
├── mise.toml             # install / format / check / test / build / run tasks
├── cmd/
│   └── agent/
│       └── main.go       # entry point (ADK full launcher)
└── internal/
    ├── config/           # typed configuration
    ├── agent/            # the Ops Copilot agent (Ch. 2)
    ├── data/             # data-access layer over incidents.db
    ├── tools/            # function tools (3.1)
    ├── skills/           # agent skills (3.2)
    ├── mcpclient/        # MCP client toolset (3.3)
    ├── memory/           # memory / RAG (3.4)
    ├── workflow/         # sequential workflow (3.5)
    ├── delegation/       # sub-agents / A2A (3.6)
    ├── actions/          # guarded mock actions (4.5)
    └── guardrails/       # safety callbacks (4.5)
```

The module path in `go.mod` doubles as the import prefix, so `internal/tools` is imported as `github.com/MLOps-Courses/agentops-open-course/agents/go/internal/tools`:

```go
// agents/go/go.mod
module github.com/MLOps-Courses/agentops-open-course/agents/go

go 1.26.0

require (
	github.com/modelcontextprotocol/go-sdk v1.6.1
	google.golang.org/adk/v2 v2.0.0
	google.golang.org/genai v1.62.0
	modernc.org/sqlite v1.53.0
)
```

Note `modernc.org/sqlite` — a pure-Go SQLite driver (no cgo), chosen so the agent later compiles into a minimal distroless image ([Chapter 6](../6. Platform/)). The `cmd/agent/main.go` entry point stays thin: it loads config, builds the agent, and hands it to the ADK launcher.

## Why `src/` in Python and `internal/` in Go?

Different idioms, the same intent — keep the public surface small and imports honest.

- **Python `src/` layout**: the package is only importable once installed, so tests exercise exactly what ships. It prevents the "works on my machine because the file happens to be in the working directory" class of bug.
- **Go `internal/`**: the compiler forbids any package outside this module from importing anything under `internal/`. The agent's building blocks are private by construction; only `cmd/agent` is the public entry point.

Both keep a **flat** module layout — one directory per capability, no deep hierarchies — which matches how the course introduces them one at a time.

## How do I install and run each track?

Every track exposes the same canonical `mise run` task vocabulary, so the commands you learn carry across languages:

=== "Python"

    ```bash
    cd agents/python
    mise run install   # uv sync
    mise run run       # adk run src/agent
    mise run web       # adk web src (developer UI)
    ```

=== "Go"

    ```bash
    cd agents/go
    mise run install        # go mod tidy
    mise run run -- console  # terminal REPL
    mise run run -- web      # ADK web UI
    ```

Under the hood these delegate to `uv` and `go` respectively, but you drive them the same way. The next pages fill in the capability modules one by one — start with [3.1. Tools](./3.1. Tools.md).
