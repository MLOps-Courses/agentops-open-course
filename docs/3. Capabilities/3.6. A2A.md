# 3.6. A2A

## What is delegation?

**Delegation** is one agent handing work to another. Instead of a single agent that tries to do everything, a **coordinator** triages the situation and, when a task needs depth, transfers control to a **specialist** that shares the same session. ADK routes to the named sub-agent, the specialist does its focused job, and the coordinator summarizes the result.

The Ops Copilot models this with two agents:

- **coordinator_agent** — the on-call coordinator: triages incidents with the read tools, and delegates deep analysis when needed.
- **diagnosis_agent** — a specialist that works out an incident's likely root cause from its runbook and service status.

This is the multi-agent pattern from [0.1. Agents](../0. Overview/0.1. Agents.md), made concrete.

## How is a sub-agent wired?

Build the specialist as a normal agent, then register it on the coordinator via `sub_agents` (Python) / `SubAgents` (Go). The coordinator's instruction tells it _when_ to delegate.

=== "Python"

    ```python
    # agents/python/src/agent/delegation.py
    # The specialist: given an incident, works out the likely cause from its runbook.
    diagnosis_agent = Agent(
        model=settings.model,
        name="diagnosis_agent",
        description="Specialist that diagnoses a specific incident using its runbook and service status.",
        instruction=(
            "You are a diagnosis specialist. Given an incident id, use get_incident for its details and "
            "runbook, get_runbook for the runbook body, and get_service_status for the service. Explain "
            "the likely root cause in a few sentences and cite the runbook."
        ),
        tools=[*ALL_TOOLS, *KNOWLEDGE_TOOLS],
    )

    # The coordinator: triages, and delegates deep diagnosis to the specialist sub-agent.
    coordinator_agent = Agent(
        model=settings.model,
        name="coordinator_agent",
        description="On-call coordinator that triages incidents and delegates diagnosis.",
        instruction=(
            "You are the on-call coordinator. Triage with list_incidents and get_service_status. When a "
            "specific incident needs a root-cause analysis, delegate to the diagnosis_agent sub-agent, "
            "then summarize its findings for the engineer."
        ),
        tools=ALL_TOOLS,
        sub_agents=[diagnosis_agent],
    )
    ```

=== "Go"

    ```go
    // agents/go/internal/delegation/delegation.go
    diagnosis, err := llmagent.New(llmagent.Config{
    	Name: "diagnosis_agent", Model: model,
    	Description: "Specialist that diagnoses a specific incident using its runbook and service status.",
    	Instruction: diagnosisInstruction, Tools: diagnosisTools,
    })
    if err != nil {
    	return nil, fmt.Errorf("building diagnosis agent: %w", err)
    }

    return llmagent.New(llmagent.Config{
    	Name: "coordinator_agent", Model: model,
    	Description: "On-call coordinator that triages incidents and delegates diagnosis.",
    	Instruction: coordinatorInstruction,
    	Tools:       readTools,
    	SubAgents:   []adkagent.Agent{diagnosis},
    })
    ```

The wiring is symmetric: a specialist agent, then a coordinator that lists it in `sub_agents` / `SubAgents`. The coordinator keeps only the triage tools; the specialist gets the full set (incident tools plus knowledge tools) because it does the deep work.

## How is delegation different from a workflow?

Both coordinate multiple agents, but they decide the control flow differently:

- A **workflow** ([3.5. Workflows](./3.5. Workflows.md)) runs a **fixed** order — triage, then diagnose, then recommend, every time.
- **Delegation** is **dynamic**: the coordinator decides _at run time_, from the situation, whether to handle the request itself or hand it to the specialist. No delegation happens if none is needed.

Use a workflow when the procedure is known and repeatable; use delegation when the coordinator must judge whether a specialist is warranted.

## What is A2A, and how does it relate to sub-agents?

Everything above runs **in-process**: coordinator and specialist are objects in the same program, sharing memory. **A2A (Agent2Agent)** is the open protocol that lets the same relationship work **across the network** — the specialist runs as its own service, and the coordinator calls it over HTTP as if it were a local sub-agent.

!!! info "AAIF project"

    **A2A** is an open **[AAIF](https://aaif.io/)** (Agentic AI Foundation) standard for agent-to-agent interoperability, alongside [MCP](./3.3. MCP.md). It lets agents built by different teams, in different languages, on different frameworks, discover and call one another through a common contract. The course revisits A2A behind the gateway in [5.3. A2A Gateway](../5. Gateway/5.3. A2A Gateway.md).

The step from in-process to networked is small because the wiring is the same:

1. **Expose** the specialist over A2A. In Python, serve it with `adk api_server`; in Go, launch it in the `a2a` launcher mode. Either way it becomes an A2A endpoint with a published agent card.
1. **Call** it remotely. The coordinator references the specialist as a **`RemoteA2aAgent`** pointing at that endpoint, instead of an in-process sub-agent object.

Because the Go agent's `full` launcher already exposes an `a2a` mode (`mise run run -- a2a`) and Python's `adk api_server` does the same, promoting a sub-agent to a networked A2A service is a deployment choice, not a rewrite. The coordinator's instruction — "delegate to the specialist, then summarize" — does not change.

## When should I split an agent into services?

Keep it in-process while it is simple; reach for networked A2A when the operational reality demands it:

- **Independent scaling or deployment**: the specialist is heavy, or owned by another team, and should ship on its own cadence.
- **Language or framework boundary**: a Go coordinator calling a Python specialist (or a third-party agent) — A2A crosses those lines that in-process sub-agents cannot.
- **Reuse across agents**: several coordinators share one specialist service.

Otherwise, in-process sub-agents are simpler, faster, and easier to test. Start there, and let A2A be the seam you cross when you actually need a network boundary.
