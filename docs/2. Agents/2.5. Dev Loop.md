# 2.5. Dev Loop

Building an agent is a tight edit-run-observe cycle: change the instruction or a tool, run the agent, watch what it does, repeat. You get three front-ends for that loop — a terminal REPL, a developer web UI, and an HTTP API server. This section is the map.

## What is the dev loop?

The loop is: edit code, run the agent, observe the event stream, adjust. The web UI makes the "observe" step visual — you see every tool call and result — which is why it is the workhorse during development. The terminal is fastest for a quick check; the API server is how you integrate the agent into something else.

The agent exposes these through `mise run` tasks, so the commands stay identical to what CI and your teammates run.

## How do you run the agent during development?

```bash
# agents/python — thin wrappers over the adk CLI (see mise.toml)
mise run web    # ADK developer web UI   (uv run adk web src)
mise run run    # terminal REPL          (uv run adk run src/agent)
```

The `adk` command-line tool discovers `root_agent` by importing the `src/agent` package (the `__init__.py` contract from [2.1. First Agent](./2.1. First Agent.md)), so you point it at the source directory and it finds the agent.

## What does the web UI give you?

The developer web UI is the reason to run it first (`adk web`). It renders each session as a chat plus a **trace**: every model call, every tool call and its arguments, every tool result, and the timing. When the Ops Copilot answers "what's wrong with checkout?", you see it call `list_incidents` and `get_service_status`, and you can read exactly what came back. That visibility is what turns a black box into something you can debug and, later, evaluate ([4.4. Evaluations](../4. Quality/4.4. Evaluations.md)) and trace to OpenTelemetry ([7.1. Tracing](../7. Observability/7.1. Tracing.md)).

The terminal REPL runs the same agent with the same tools; it just prints the loop as text. Use it when you want a fast smoke test without a browser.

## What run modes exist?

The `adk` CLI has one subcommand per mode:

- **`adk web src`** — the developer web UI (the `web` task).
- **`adk run src/agent`** — the terminal REPL (the `run` task).
- **`adk api_server src`** — an HTTP API server, for wiring the agent into another service or a test harness. Not a mise task; run it directly with `uv run adk api_server src`.

!!! note "Every mode needs a provider key"

    All of these call a real Gemini model, so `.env` must hold `GOOGLE_API_KEY` (plus `GOOGLE_GENAI_USE_VERTEXAI=true` for a Vertex AI express key). See [2.2. Models](./2.2. Models.md).

## Which mode should you use when?

A simple rule of thumb:

1. **`adk web`** — default for building and debugging; the trace view is worth it.
1. **`adk run`** — a quick REPL smoke test after a small change.
1. **`adk api_server`** — integrating the agent into another program, a script, or a test.
1. **`adk api_server --a2a`** — exposing the agent to other agents; the subject of [3.6. A2A](../3. Capabilities/3.6. A2A.md).

That completes the first agent: you can define it, model it, instruct it, give it sessions, and run it every way the frameworks offer. [Chapter 3](../3. Capabilities/) gives the Ops Copilot its real powers — tools, skills, MCP, memory, workflows, and delegation.
