# 2.6. Dev Loop

Building an agent is a tight edit-run-observe cycle: change the instruction or a tool, run the agent, watch what it does, repeat. Both tracks give you the same three front-ends for that loop ‚Äî a terminal REPL, a developer web UI, and an HTTP API server ‚Äî reached slightly differently. This section is the map.

## What is the dev loop?

The loop is: edit code, run the agent, observe the event stream, adjust. The web UI makes the "observe" step visual ‚Äî you see every tool call and result ‚Äî which is why it is the workhorse during development. The terminal is fastest for a quick check; the API server is how you integrate the agent into something else.

Both tracks expose these through `mise run` tasks, so the commands stay identical to what CI and your teammates run.

## How do you run the agent during development?

=== "Python"

    ```bash
    # agents/python ‚Äî thin wrappers over the adk CLI (see mise.toml)
    mise run web    # ADK developer web UI   (uv run adk web src)
    mise run run    # terminal REPL          (uv run adk run src/agent)
    ```

    Python ships a standalone `adk` command-line tool. It discovers `root_agent` by importing the `src/agent` package (the `__init__.py` contract from [2.1](./2.1. First Agent (Python).md)), so you point it at the source directory and it finds the agent.

=== "Go"

    ```bash
    # agents/go ‚Äî arguments after `--` pass straight to the launcher (see mise.toml)
    mise run run -- web webui   # ADK web UI (browser)
    mise run run -- console     # terminal REPL
    ```

    Go has no separate CLI: your `main` embeds the launcher. `mise run run` wraps `go run ./cmd/agent`, and everything after `--` is the launcher's mode arguments. The `full.NewLauncher()` from [2.2](./2.2. First Agent (Go).md) bundles every mode into that one binary ‚Äî `web` is a server whose sub-servers (`webui`, `api`, `a2a`) you name after it.

## What does the web UI give you?

The developer web UI is the reason to run it first (`adk web` in Python, `web webui` in Go). It renders each session as a chat plus a **trace**: every model call, every tool call and its arguments, every tool result, and the timing. When the Ops Copilot answers "what's wrong with checkout?", you see it call `list_incidents` and `get_service_status`, and you can read exactly what came back. That visibility is what turns a black box into something you can debug and, later, evaluate ([4.4. Evaluations](../4. Quality/4.4. Evaluations.md)) and trace to OpenTelemetry ([7.1. Tracing](../7. Observability/7.1. Tracing.md)).

The terminal REPL runs the same agent with the same tools; it just prints the loop as text. Use it when you want a fast smoke test without a browser.

## What run modes exist?

Each track offers the same set of front-ends, exposed a little differently:

=== "Python"

    The `adk` CLI has one subcommand per mode:

    - **`adk web src`** ‚Äî the developer web UI (the `web` task).
    - **`adk run src/agent`** ‚Äî the terminal REPL (the `run` task).
    - **`adk api_server src`** ‚Äî an HTTP API server, for wiring the agent into another service or a test harness. Not a mise task; run it directly with `uv run adk api_server src`.

=== "Go"

    The `full` launcher selects a mode from its keywords, all in one binary. The two top-level keywords are `console` and `web`; `web` is a server you pair with one or more sub-servers:

    - **`console`** ‚Äî the terminal REPL (the default).
    - **`web webui`** ‚Äî the developer web UI.
    - **`web api`** ‚Äî an HTTP API server.
    - **`web a2a`** ‚Äî an Agent2Agent endpoint, so other agents can call this one.

    Run any of them with `mise run run -- <mode>` (e.g. `mise run run -- web webui`). `web` on its own errors ‚Äî it needs a sub-server keyword. To ship a standalone binary, `mise run build` produces `bin/agent`, which you invoke as `./bin/agent <mode>`.

!!! note "Every mode needs a provider key"

    All of these call a real Gemini model, so `.env` must hold `GOOGLE_API_KEY` (plus `GOOGLE_GENAI_USE_VERTEXAI=true` for a Vertex AI express key). See [2.3. Models](./2.3. Models.md).

## Which mode should you use when?

A simple rule of thumb:

1. **`adk web` / `web webui`** ‚Äî default for building and debugging; the trace view is worth it.
1. **`adk run` / `console`** ‚Äî a quick REPL smoke test after a small change.
1. **`adk api_server` / `web api`** ‚Äî integrating the agent into another program, a script, or a test.
1. **`web a2a`** (Go) / **`adk api_server --a2a`** (Python) ‚Äî exposing the agent to other agents; the subject of [3.6. A2A](../3. Capabilities/3.6. A2A.md).

That completes the first agent: you can define it, model it, instruct it, give it sessions, and run it every way the frameworks offer. [Chapter 3](../3. Capabilities/) gives the Ops Copilot its real powers ‚Äî tools, skills, MCP, memory, workflows, and delegation.
