# 2.1. First Agent

This is the hero build. By the end you have the reference agent — the **Ops Copilot** — running in both a browser UI and your terminal, answering real questions over a bundled dataset. It assumes you finished [Chapter 1](../1. Setup/) (mise, uv, and a provider key in `.env`).

## What are you building?

The **Ops Copilot** is an on-call assistant that helps engineers triage and resolve incidents for a fictional platform, using a 100% local, bundled dataset — no external services. In this chapter it already carries its tools; you build the agent that ties a Gemini model, an on-call persona, and those tools together, then run it.

The reference agent lives in `agents/python/`. Its layout mirrors what any ADK project needs:

- `src/agent/config.py` — typed settings (which model, where the data is).
- `src/agent/agent.py` — the agent itself (`root_agent`).
- `src/agent/__init__.py` — the package the `adk` CLI imports to find the agent.

## How does the agent find its configuration?

Configuration is typed and loaded from the environment, so a missing or malformed value fails fast instead of surfacing as a confusing runtime error. The course uses `pydantic-settings`:

```python
# agents/python/src/agent/config.py
from pathlib import Path

from pydantic_settings import BaseSettings, SettingsConfigDict

# The bundled dataset lives at agents/data. This file is
# agents/python/src/agent/config.py, so the repo's agents/ dir is three parents up.
_DEFAULT_DATA_DIR = Path(__file__).resolve().parents[3] / "data"


class Settings(BaseSettings):
    """Agent settings. Native Gemini in Ch. 2-4; other providers arrive via agentgateway in Ch. 5."""

    model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8", env_prefix="AGENT_", extra="ignore")

    # Set an explicit model — ADK defaults churn. Override with AGENT_MODEL.
    model: str = "gemini-3.5-flash"

    # Where the bundled Ops Copilot dataset lives. Override with AGENT_DATA_DIR (e.g. in containers).
    data_dir: Path = _DEFAULT_DATA_DIR


settings = Settings()
```

Two things worth noting:

- The **model is set explicitly** (`gemini-3.5-flash`). ADK's built-in defaults change between releases, so pinning it keeps behavior reproducible. You can still override it with the `AGENT_MODEL` environment variable, thanks to the `AGENT_` prefix.
- `settings` is a **singleton** imported everywhere else, so the whole agent reads one consistent configuration.

## How do you define the agent?

The agent is a single declaration: a model, an identity, an instruction, and tools.

```python
# agents/python/src/agent/agent.py
from google.adk import Agent

from .actions import ACTION_TOOLS
from .config import settings
from .guardrails import validate_actions
from .memory import KNOWLEDGE_TOOLS
from .tools import ALL_TOOLS

root_agent = Agent(
    model=settings.model,
    name="agentops_agent",
    description="An on-call Ops Copilot that triages and resolves incidents from a local dataset.",
    instruction=INSTRUCTION,
    tools=[*ALL_TOOLS, *KNOWLEDGE_TOOLS, *ACTION_TOOLS],
    before_tool_callback=validate_actions,
)
```

`Agent` is ADK's public alias for the `LlmAgent` from [2.0. Concepts](./2.0. Concepts.md). Its arguments map one-to-one onto the ideas you already know:

- `model` — the Gemini model from `settings`, wired up in [2.2. Models](./2.2. Models.md).
- `name` and `description` — the agent's identity, used by the UI and by other agents over A2A.
- `instruction` — the persona and operating rules (below), detailed in [2.3. Instructions](./2.3. Instructions.md).
- `tools` — the incident, knowledge, and action tools the model may call, built in [Chapter 3](../3. Capabilities/).
- `before_tool_callback` — a guardrail that runs before any tool, so state-changing actions need approval. Covered in [4.5. Guardrails](../4. Quality/4.5. Guardrails.md).

!!! note "The tools are already written"

    `ALL_TOOLS`, `KNOWLEDGE_TOOLS`, and `ACTION_TOOLS` come from other modules in the package. You do not need to understand them yet — Chapter 3 builds each one. For now they make the agent genuinely useful so you can watch the loop call them.

## Why is the instruction its own string?

The persona and rules are kept as an explicit constant, not buried in the `Agent` call, so behavior is reproducible and easy to evaluate:

```python
# agents/python/src/agent/agent.py
# The persona and operating rules. Kept explicit so behavior is reproducible and evaluable.
INSTRUCTION = """\
You are the Ops Copilot, an on-call assistant for a fictional online platform.
You help engineers triage and resolve incidents quickly and safely.

Operating rules:
- Always ground your answers in the tools. Never invent incidents, services, or statuses.
- When asked about incidents or a service, call the matching tool and report exactly what it returns.
- To recommend a fix, consult the runbooks: an incident carries a `runbook` slug — fetch it with
  `get_runbook`, or use `search_runbooks` to find guidance by symptom. Cite the runbook you used.
- Taking an action (restart_service, resolve_incident) changes state and needs human approval —
  propose it, and only call the tool when the engineer asks you to. Report the audit result.
- Refer to incidents by id (e.g. INC-001) and services by name (e.g. checkout).
- Be concise and actionable: lead with the answer, then the key details.
- If a tool returns an error or no data, say so plainly instead of guessing.
"""
```

The first rule — _always ground your answers in the tools_ — is what makes this an agent and not a chatbot: it must look things up rather than guess. [2.3. Instructions](./2.3. Instructions.md) unpacks why each rule is written the way it is.

## How does the `adk` CLI find the agent?

The `adk` command-line tool discovers your agent by importing the package and reading a module named `agent` that exposes `root_agent`. The package `__init__.py` re-exports it:

```python
# agents/python/src/agent/__init__.py
"""The AgentOps Open Course reference agent.

The `adk` CLI (`adk web` / `adk run` / `adk api_server`) discovers `root_agent`
by importing this package, which re-exports the `agent` module below.
"""

from . import agent

__all__ = ["agent"]
```

That is the whole contract: define `root_agent` and make it importable. No registration, no manifest.

## How do you run it in the browser?

The friendliest way to try the agent is the ADK developer web UI. From the agent directory:

```bash
cd agents/python
mise run web        # ADK developer web UI  (uv run adk web src)
```

`mise run web` is a thin wrapper around `uv run adk web src` (see `agents/python/mise.toml`). It launches a local web server; open the printed URL, pick the `agentops_agent`, and chat. Try:

> _List the open incidents, then tell me what's wrong with checkout._

Watch the trace panel: the model calls `list_incidents`, then `get_service_status`, then answers from the results. That live event stream is the agentic loop from [2.0. Concepts](./2.0. Concepts.md), rendered.

!!! warning "You need a provider key"

    `web` and `run` call a real Gemini model, so they need credentials in `.env` (`GOOGLE_API_KEY`, and `GOOGLE_GENAI_USE_VERTEXAI=true` for a Vertex AI express key). If you skipped it, set that up in [1.4. Providers](../1. Setup/1.4. Providers.md) and [2.2. Models](./2.2. Models.md) first.

## How do you run it in the terminal?

For a quick REPL without a browser, run it in the terminal:

```bash
mise run run        # run in the terminal  (uv run adk run src/agent)
```

`mise run run` wraps `uv run adk run src/agent`. You type messages, the agent replies, and tool calls print inline. It is the fastest way to smoke-test a change to the instruction or a tool. The full set of run modes — including an HTTP API server — is the subject of [2.5. Dev Loop](./2.5. Dev Loop.md).
