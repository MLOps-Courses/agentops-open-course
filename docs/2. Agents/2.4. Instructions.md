# 2.4. Instructions

The instruction is the agent's system prompt: the persona and rules the model follows on every turn. It is the single biggest lever on behavior, and the easiest to get wrong. This section reads the Ops Copilot's real instruction line by line and shows how to constrain the output.

## What is a system instruction?

The **instruction** is fixed guidance prepended to every request the agent sends the model. Unlike the user's message — which changes each turn — the instruction is constant, so it is where you set identity, tone, and hard rules. In ADK you pass it when you build the agent: `instruction=` in Python, `Instruction:` in Go.

Both tracks keep it in a named constant rather than inline, so behavior is **reproducible and evaluable**: the exact prompt is versioned in the repo, diffable in review, and asserted against in the evals of [4.4. Evaluations](../4. Quality/4.4. Evaluations.md).

## What does the Ops Copilot's instruction say?

Here is the real instruction, verbatim from each track. The text is identical; only the string syntax differs (a Python triple-quoted string versus a Go raw string literal).

=== "Python"

    ```python
    # agents/python/src/agent/agent.py
    # The persona and operating rules. Kept explicit so behavior is reproducible and evaluable.
    INSTRUCTION = """\
    You are the Ops Copilot, an on-call assistant for a fictional online platform.
    You help engineers triage and resolve incidents quickly and safely.

    Operating rules:
    - Always ground your answers in the tools. Never invent incidents, services, or statuses.
    - When asked about incidents or a service, call the matching tool and report exactly what it returns.
    - To recommend a fix, consult the runbooks: an incident carries a `runbook` slug — fetch it with
      `get_runbook`, or use `search_runbooks` to find guidance by symptom. Cite the runbook you used.
    - Taking an action (restart_service, resolve_incident) changes state and needs human approval —
      propose it, and only call the tool when the engineer asks you to. Report the audit result.
    - Refer to incidents by id (e.g. INC-001) and services by name (e.g. checkout).
    - Be concise and actionable: lead with the answer, then the key details.
    - If a tool returns an error or no data, say so plainly instead of guessing.
    """
    ```

=== "Go"


## Why does the first rule matter most?

_"Always ground your answers in the tools. Never invent incidents, services, or statuses."_

**Grounding** means the agent bases its answer on data it fetched, not on the model's training memory. This is the line between a useful ops assistant and a dangerous one: an on-call engineer acts on what the agent says, so a plausible-but-invented incident id or service status is worse than "I don't know."

The instruction reinforces grounding several ways: call the matching tool and _report exactly what it returns_; cite the runbook you used; and _if a tool returns an error or no data, say so plainly instead of guessing_. Grounding is a prompt-level habit here; it becomes an enforced guarantee with retrieval in [3.4. Memory](../3. Capabilities/3.4. Memory.md) and guardrails in [4.5. Guardrails](../4. Quality/4.5. Guardrails.md).

## What makes a good operating rule?

Each rule in the list is doing a specific job worth copying:

- **Name the tools and when to use them** — "call the matching tool", "fetch it with `get_runbook`" — so the model routes to the right action instead of freelancing.
- **Gate risky actions** — "restart_service / resolve_incident … needs human approval" — the prompt-level half of human-in-the-loop, backed by a guardrail callback in code.
- **Fix the vocabulary** — refer to incidents by id (`INC-001`), services by name (`checkout`) — so outputs are consistent and evaluable.
- **Set the shape of a good answer** — "concise and actionable: lead with the answer, then the key details."

Rules are concrete and testable. Vague guidance ("be helpful") gives the model nothing to comply with; specific, checkable rules do.

## Why keep the instruction in code, not a database?

Because the prompt _is_ behavior. Keeping it in the source tree means it is version-controlled, reviewed in pull requests, and pinned alongside the model version — the reproducibility discipline of [7.0. Reproducibility](../7. Observability/7.0. Reproducibility.md). A prompt edited live in a UI is an unversioned production change; a prompt in `agent.py` / `agent.go` is a diff.

## How do you get structured output?

Sometimes you want the agent to return a typed object — a JSON incident summary — rather than prose, so downstream code can consume it. ADK supports this with an output schema on the agent:

=== "Python"

    ```python
    # Constrain the final response to a schema (pydantic model / JSON schema).
    Agent(model=settings.model, name="agentops_agent", instruction=INSTRUCTION, output_schema=IncidentSummary)
    ```

=== "Go"

    ```go
    // Constrain the final response to a genai.Schema.
    llmagent.New(llmagent.Config{Name: Name, Model: model, Instruction: Instruction, OutputSchema: incidentSummarySchema})
    ```

The Ops Copilot does not set one — it answers conversationally and acts through tools — so this is here as a pointer. Reach for `output_schema` / `OutputSchema` when a caller needs machine-readable results; otherwise keep the instruction's plain-language shape. Tools, which let the agent _act_ rather than just format text, are the subject of [3.1. Tools](../3. Capabilities/3.1. Tools.md).
